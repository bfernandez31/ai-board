## Implementation Choices Analysis

### Decision Point 1: Pagination Strategy
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-172 | Offset-based with limit | ✅ Simpler | No cursor tracking, may shift |
| AIB-177 | Cursor-based with base64 | ✅ Robust | More complex, handles expiry |

**Winner**: AIB-177 - Cursor-based pagination is more robust for real-time feeds with continuous updates.

### Decision Point 2: Event Type Coverage
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-172 | 6 event types (ticket, comment, job start/complete/fail, stage) | ✅ | Core events covered |
| AIB-177 | 8 event types (adds pr_created, preview_deployed) | ✅ | More derived events |

**Winner**: AIB-177 - Additional derived events provide richer activity context.

### Decision Point 3: Actor Management
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-172 | Separate actors map, returned with response | ✅ | Deduplication, normalized |
| AIB-177 | Actor embedded in each event, type discriminated | ✅ | Simpler parsing |

**Winner**: AIB-172 - Normalized actors map reduces payload size and enables efficient lookup.

### Decision Point 4: UI Scroll Preservation
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-172 | `scrollPositionRef` with `requestAnimationFrame` | ✅ | Smooth UX during polling |
| AIB-177 | State-based merge without scroll handling | ⚠️ | May jump during updates |

**Winner**: AIB-172 - Explicit scroll preservation provides better UX during real-time updates.

### Decision Point 5: Component Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-172 | Separate event icons + event item components | ✅ | Better separation, reusable |
| AIB-177 | Single activity-item component | ✅ | Simpler, less abstraction |

**Winner**: AIB-172 - Better component separation follows shadcn/ui patterns.

### Decision Point 6: State Management
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-172 | TanStack Query + local pagination state | ✅ | Proper server state |
| AIB-177 | TanStack Query infinite query | ✅ | Built-in pagination |

**Winner**: Tie - Both use TanStack Query appropriately per constitution.

## Ranking & Recommendation

### 1. **AIB-172** - Best code (score: 88%)
**Why it wins**:
- Superior UX with scroll position preservation during polling
- Better component separation (event icons, event item as separate components)
- Normalized actors map reduces response payload
- More comprehensive test coverage (1728 vs 1476 lines)
- Cleaner utility separation with `activity-events.ts` and `activity-event-icons.tsx`

**Areas where AIB-177 is better**:
- Cursor-based pagination is more robust
- Additional event types (pr_created, preview_deployed)
- cursorExpired handling for stale pagination

### 2. **AIB-177** - Good implementation (score: 85%)
**Why it lost**:
- No scroll preservation during polling updates
- Fewer component separations
- Slightly fewer test lines
- More jobs/cost for similar outcome

**Strengths to consider cherry-picking**:
- Cursor-based pagination approach
- pr_created and preview_deployed event types
- cursorExpired toast notification pattern
