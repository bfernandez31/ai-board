## Implementation Choices Analysis

### Decision Point 1: API Architecture

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-147 | Extended `/transition` endpoint with inline logic | ⚠️ Mixed concerns | Reuses existing route, but couples close logic with transition state machine |
| AIB-148 | Dedicated `/close` endpoint | ✅ Single responsibility | Clean separation, easier to test, but creates parallel endpoint |

**Winner**: **AIB-148** - Follows Single Responsibility Principle. Close operation has distinct behavior (PR closure, no workflow dispatch) that justifies dedicated endpoint. Cleaner API contract.

### Decision Point 2: GitHub Integration

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-147 | `closePRsOnly()` - Simple closure | ✅ Idempotent | Minimal, focused function. Returns `{prsClosed: number}` |
| AIB-148 | `closePRsForBranch()` - Detailed tracking | ✅ Idempotent + metrics | Returns `{prsClosed, prsAlreadyClosed}`. Tracks already-closed PRs separately |

**Winner**: **AIB-148** - Better observability. Tracking already-closed PRs helps debugging idempotency. More robust error handling (404/422 distinction).

### Decision Point 3: Client-Side State Management

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-147 | `useCloseTicket()` custom hook | ✅ TanStack Query | Optimistic updates, automatic rollback, cache invalidation. Clean separation |
| AIB-148 | Inline `useMutation` in Board component | ❌ Mixed concerns | Direct mutation without dedicated hook. Less reusable |

**Winner**: **AIB-147** - Custom hook (`useCloseTicket`) is properly abstracted, testable, and reusable. Better separation of concerns.

### Decision Point 4: Testing Strategy

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-147 | No tests | ❌ Violates Principle III | Feature not verifiable |
| AIB-148 | 3 test files (535 lines) | ✅ Full coverage | Integration (close flow), Component (modal UI), Unit (GitHub lib) |

**Winner**: **AIB-148** - Tests verify:
1. **Integration** (248 lines): VERIFY→CLOSED transition, active job validation, PR closure
2. **Component** (116 lines): Close confirmation modal interactions
3. **Unit** (171 lines): GitHub PR closure logic

AIB-147 has **ZERO tests**, violating Constitution Principle III (Non-Negotiable).

The `/transition` endpoint now handles two distinct operations:
1. **Stage transitions**: Workflow dispatch, job creation
2. **Close operation**: No workflow, PR closure only

This violates Single Responsibility Principle. While the code is clean, mixing concerns makes future maintenance harder.
