## Core Entities

### Ticket (Existing - Modified)

**Description**: A work item tracked on the kanban board

**Schema Changes Required**:
```prisma
model Ticket {
  id          Int      @id @default(autoincrement())
  title       String   @db.VarChar(100)        // CHANGE: from VarChar(500) to VarChar(100)
  description String   @db.VarChar(1000)       // CHANGE: from Text? (nullable) to VarChar(1000) (required)
  stage       Stage    @default(IDLE)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@index([stage])
  @@index([updatedAt])
}

enum Stage {
  IDLE      // Default for new tickets
  PLAN
  BUILD
  REVIEW
  SHIPPED
  ERRORED
}
```

**Field Specifications**:

| Field | Type | Constraints | Notes |
|-------|------|-------------|-------|
| `id` | Int | Primary key, auto-increment | Generated by database |
| `title` | String | Required, 1-100 chars, pattern validation | User-provided, validated by Zod |
| `description` | String | Required, 1-1000 chars, pattern validation | User-provided, validated by Zod |
| `stage` | Stage | Required, default IDLE | System-managed, always IDLE for new tickets |
| `createdAt` | DateTime | Required, auto-generated | System timestamp |
| `updatedAt` | DateTime | Required, auto-updated | System timestamp |

**Breaking Change**: Yes - existing tickets with null descriptions will fail migration
- **Mitigation**: Add default value in migration or require data backfill before migration

## Validation Rules

### Client-Side Validation (Zod Schema)

**Location**: `/lib/validations/ticket.ts`

```typescript
import { z } from "zod";

// Pattern for alphanumeric + basic punctuation (periods, commas, hyphens, spaces, question marks, exclamation points)
const ALPHANUMERIC_PUNCTUATION = /^[a-zA-Z0-9\s.,?!\-]+$/;

export const createTicketSchema = z.object({
  title: z
    .string()
    .min(1, "Title is required")
    .max(100, "Title must be 100 characters or less")
    .regex(
      ALPHANUMERIC_PUNCTUATION,
      "Title can only contain letters, numbers, and basic punctuation (. , ? ! -)"
    )
    .trim(),
  description: z
    .string()
    .min(1, "Description is required")
    .max(1000, "Description must be 1000 characters or less")
    .regex(
      ALPHANUMERIC_PUNCTUATION,
      "Description can only contain letters, numbers, and basic punctuation (. , ? ! -)"
    )
    .trim(),
});

// Infer TypeScript type from schema
export type CreateTicketInput = z.infer<typeof createTicketSchema>;
```

**Validation Behavior**:
- **Real-time**: Validate on blur and on change (debounced)
- **On Submit**: Full validation before API call
- **Error Display**: Show errors inline below each field
- **Button State**: Disable Create button when any field is invalid

### Server-Side Validation

**Location**: `/app/api/tickets/route.ts`

```typescript
import { createTicketSchema } from "@/lib/validations/ticket";

// In POST handler
const body = await req.json();
const result = createTicketSchema.safeParse(body);

if (!result.success) {
  return Response.json(
    { error: "Invalid input", details: result.error.flatten() },
    { status: 400 }
  );
}

// Use result.data for Prisma insert
```

**Validation Order**:
1. Parse request body as JSON
2. Validate with Zod schema (same rules as client)
3. If invalid: Return 400 with structured errors
4. If valid: Proceed to database insert

## State Transitions

### Modal State Machine

```
[Closed] --click "+ New Ticket"--> [Open/Empty]
[Open/Empty] --user types--> [Open/Dirty]
[Open/Dirty] --click Create (valid)--> [Submitting]
[Submitting] --success--> [Closed] + ticket added to IDLE
[Submitting] --error--> [Open/Error]
[Open/*] --click Cancel--> [Closed]
[Open/*] --click outside--> [Closed]
[Open/*] --press Escape--> [Closed]
```

**State Variables**:
- `open: boolean` - Modal visibility
- `formData: TicketFormState` - Current form values
- `errors: TicketFormErrors` - Validation errors
- `isSubmitting: boolean` - Loading state during API call

### Form Validation State

```
[Empty] --user types--> [Typing]
[Typing] --debounce 300ms--> [Validating]
[Validating] --pass--> [Valid]
[Validating] --fail--> [Invalid] + show errors
[Invalid] --user corrects--> [Typing]
[Valid] --user submits--> [Submitting]
```

**Validation Timing**:
- **onChange**: Debounced validation (300ms) for real-time feedback
- **onBlur**: Immediate validation when field loses focus
- **onSubmit**: Full validation before API call

## Data Flow

### 1. User Opens Modal
```
User clicks "+ New Ticket"
  → NewTicketButton calls setOpen(true)
  → NewTicketModal renders with empty form
  → Focus automatically moves to title field
```

### 2. User Fills Form
```
User types in title field
  → onChange updates formData.title
  → Debounced validation after 300ms
  → If invalid: errors.title = "error message"
  → If invalid: Create button disabled

User types in description field
  → Same validation flow as title
```

### 3. User Submits Form
```
User clicks Create button
  → Full validation runs
  → If invalid: Show errors, don't submit
  → If valid: setIsSubmitting(true)
  → POST /api/tickets with formData
  → API validates with Zod
  → API creates ticket in database
  → API returns 201 + ticket data
  → Client closes modal
  → Client triggers onTicketCreated callback
  → Board component refreshes to show new ticket
```

### 4. Error Handling
```
Network error or timeout (>15s)
  → Catch error in try-catch
  → setErrors({ submit: "Unable to create ticket. Please try again." })
  → setIsSubmitting(false)
  → Modal stays open, user can retry

Validation error (400 response)
  → Parse error details
  → setErrors with field-specific errors
  → setIsSubmitting(false)
  → Modal stays open, show errors
```
