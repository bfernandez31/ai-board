# ai-board Vision

## Target Version Summary
- SaaS workspace where teams import or create projects, connect GitHub repos, and manage ticket workflows end-to-end.
- Spec-kit (Claude-driven) generates ticket documentation automatically; users review, edit, and approve specs directly inside the app.
- Tickets progress through gated steps (spec review, implementation, verification) with AI agents drafting code branches and syncing back to GitHub.
- Specs live on dedicated GitHub branches so teams retain the full history while editing through the web UI—no manual git commands required.
- MVP focuses on an inception use case: ai-board itself serves as the demo project, allowing live refresh of spec updates and full control over the repo workflow before onboarding external repos.
- Overall MVP emphasizes local-first development, seamless GitHub integration, and a lean deployment story (Vercel + managed Postgres + Claude provider).

## Workflow Overview

### Stage Progression
Tickets flow through 6 distinct stages with automated or manual transitions:

```
INBOX → SPECIFY → PLAN → BUILD → VERIFY → SHIP
```

#### Stage Definitions

**INBOX**
- Initial landing zone for new tickets
- Users create ticket with title and description
- Choose automation mode: **Auto** or **Manual**
- Edit ticket details before proceeding
- Drag-drop to SPECIFY to begin spec-kit workflow

**SPECIFY** 🤖
- Triggers spec-kit `/specify` command execution
- Generates formal specification in `specs/<ticket-id>/spec.md`
- Creates feature branch: `feature/ticket-<id>`
- **Auto Mode**: Automatically transitions to PLAN after spec generation
- **Manual Mode**: Waits for user validation (drag-drop to PLAN)
- Commits spec to GitHub branch

**PLAN** 📋
- Executes `/plan` and `/task` commands sequentially
- Generates `plan.md` (technical design) and `tasks.md` (action items)
- **Auto Mode**: Automatically transitions to BUILD after task generation
- **Manual Mode**: User reviews plan/tasks, edits if needed, then drags to BUILD
- Commits plan artifacts to feature branch

**BUILD** ⚙️
- Executes `/implement` command
- AI agent implements code changes based on tasks
- Runs tests and type-checking
- **Auto Mode**: Automatically transitions to VERIFY after implementation
- **Manual Mode**: User reviews code changes, runs local tests, then drags to VERIFY
- Commits implementation to feature branch

**VERIFY** ✅
- Manual validation stage (no automated spec-kit command)
- User reviews complete implementation
- Runs E2E tests, checks quality
- **Auto Mode**: Claude Code opens/reviews PR and merges to `main` once checks pass
- **Manual Mode**: Team opens PR, performs review, and merges via GitHub
- Drag to SHIP when ready to merge

**SHIP** 🚀
- Stage flips automatically when PR merge lands on `main`
- CI pipeline deploys on merge to main branch
- Ticket marked as completed
- Final stage (no further transitions)

### Automation Modes

#### Auto Mode
- **Trigger**: User selects "Auto" when creating ticket
- **Behavior**: Each spec-kit command automatically triggers next stage transition
- **Flow**: SPECIFY → auto → PLAN → auto → BUILD → auto → VERIFY (Claude opens/reviews PR) → auto → SHIP (after merge)
- **Use Case**: Rapid prototyping, trusted workflows, low-risk changes
- **Commits**: Automatic commit/push after each command execution
- **PR Handling**: Claude Code opens PR in VERIFY stage, reviews, and merges once checks pass

#### Manual Mode
- **Trigger**: User selects "Manual" when creating ticket (default)
- **Behavior**: User must drag-drop ticket to advance after each command
- **Flow**: SPECIFY → wait → PLAN → wait → BUILD → wait → VERIFY (human opens/reviews PR) → wait → SHIP (after merge)
- **Use Case**: Critical features, learning spec-kit, quality validation needed
- **Commits**: Same automatic commits, but user controls when to proceed
- **PR Handling**: Team opens PR during VERIFY stage and completes review/merge in GitHub
- **Benefit**: Review and edit generated artifacts (spec.md, plan.md, tasks.md) before continuing

### Git Integration

**Branch Strategy**
- Each ticket gets dedicated feature branch: `feature/ticket-<id>`
- Branch created during SPECIFY stage
- All spec-kit outputs committed to this branch
- PR created during VERIFY stage
- Auto Mode: Claude Code opens, reviews, and merges PR upon approval
- Manual Mode: Human reviewers open PR, review, and merge via GitHub UI

**Commit Strategy**
- Automatic commits after each spec-kit command execution
- Commit messages: `feat(ticket-<id>): <command> - <description>`
- Examples:
  - `feat(ticket-42): specify - add user authentication spec`
  - `feat(ticket-42): plan - design authentication architecture`
  - `feat(ticket-42): implement - add JWT auth middleware`

**File Structure**
```
specs/
  <ticket-id>/
    spec.md         # Generated by /specify
    plan.md         # Generated by /plan
    tasks.md        # Generated by /task

<project-root>/
  <implementation-files>  # Generated by /implement
```

## Spec-kit Execution Flow

### Implementation Strategies
Two approaches documented in detail:
- **MVP Approach**: GitHub Actions (see `mvp-quickstart.md`)
- **Target Approach**: Docker containers (see `architecture-target.md`)

### Execution Trigger
- **User Action**: Drag ticket to SPECIFY/PLAN/BUILD stage
- **Auto Mode**: Automatic trigger after previous command completes
- **Manual Mode**: Explicit user drag-drop required
- **Backend**: API route creates job record with pending status

### Execution Process
1. **Job Creation**: Create job (ticketId, command, status: pending, branch: feature/ticket-<id>)
2. **Environment Launch**: Start GitHub Action workflow or spawn Docker container
3. **Repository Checkout**: Clone and checkout feature branch
4. **Spec-kit Execution**: Run command with Claude Code CLI
   - `/specify`: Generate specification from ticket details
   - `/plan` + `/task`: Create technical design and task breakdown
   - `/implement`: Generate code implementation
5. **Git Commit**: Commit generated files with descriptive message
6. **Git Push**: Push commits to feature branch on GitHub
7. **Status Update**: Update job record (status: completed/failed, logs, duration)
8. **Environment Cleanup**: Destroy container or complete GitHub Action

### Requirements
- **Claude Code CLI**: Installed in execution environment
- **spec-kit**: Python package with /specify, /plan, /task, /implement commands
- **Git**: For cloning, committing, pushing
- **Node.js**: Required by Claude Code
- **Python 3.11+**: Required by spec-kit
- **Anthropic API Key**: For Claude API access
- **GitHub Token**: For repository operations

### Performance & Cost
- **Execution Time**:
  - `/specify`: 10-30s
  - `/plan` + `/task`: 15-45s
  - `/implement`: 30-120s
- **Claude API Cost**: $0.01-0.10 per command (varies with context size)
- **Infrastructure Cost**:
  - MVP (GitHub Actions): $0 (free tier: 2000 min/month)
  - Target (Docker): $20-50/month (Fly.io Machines + Redis)

## Deployment Notes
- **Local Development**: Node.js 22.20.0 LTS, Next.js + Prisma with Dockerized Postgres, specs stored in GitHub branches (starting with the ai-board repo), Anthropic API key in `.env`.
- **Production MVP**: Deploy frontend/backend on Vercel (Node.js 22.20.0 LTS), use managed Postgres (Neon/Supabase), GitHub Personal Access Token for repo operations, Claude credentials via environment variables.
- **Switching Claude Providers**: Only requires updating API client configuration (supports Bedrock, Vertex, etc.)

## Auto-Deployment Strategy
- **Phase 1 (MVP)**: Rely on existing CI/CD—Claude opens/merges PRs in Auto Mode; human reviewers merge in Manual Mode; both trigger deployment pipelines
- **Phase 2**: Opt-in connectors for Vercel, GitHub Actions deployment hooks
- **Phase 3**: Custom webhooks/API callbacks for Netlify, Render, Kubernetes
- **Phase 4**: Deeper integrations for selected platforms when demand justifies
