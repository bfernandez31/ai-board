## Implementation Choices Analysis

### Decision Point 1: Filter domain model
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-289 | `AnalyticsFilters` with constrained `range/outcome/agent` unions and shared defaults | Yes | Slightly more code, much better coherence |
| AIB-287 | Separate `range/status/agent` params with `agent: string | null` | Partial | Smaller patch, weaker validation and looser contract |

**Best**: AIB-289. It keeps the API and UI aligned and rejects unsupported agents up front.

### Decision Point 2: Agent filtering semantics
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-289 | Filters tickets by effective agent (`ticket.agent` or project `defaultAgent`) and derives agent options from completed tickets/jobs | Yes | More query logic, matches product semantics |
| AIB-287 | Filters jobs by `job.model` and lists agents from distinct job models only | No | Simpler, but can misreport data for default-agent tickets |

**Best**: AIB-289. AIB-287 can exclude tickets whose agent is inherited from project defaults and can show inconsistent totals.

### Decision Point 3: Empty and filtered states
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-289 | Preserves completion cards and supplies filter-aware empty messages per chart | Yes | Slightly more UI branching |
| AIB-287 | Short-circuits to a generic empty state when `hasData` is false | Partial | Simpler, but hides useful completion metrics |

**Best**: AIB-289. It preserves meaningful filtered analytics instead of collapsing the whole dashboard.

## Ranking & Recommendation

### 1. **AIB-289** - Best code (91%)
Why it wins:
- Strong typed filter model across API, queries, and dashboard state
- Correct agent semantics using ticket/default-agent ownership instead of raw job model strings
- Adds integration and component coverage for filter behavior and invalid input handling
- Handles no-match states without discarding completion metrics

### 2. **AIB-287** - (66%)
Why it lost:
- No automated tests
- Agent filtering is semantically weaker and can diverge from ticket ownership/default-agent behavior
- Workflow distribution and velocity are not agent-filtered, so filtered dashboards can display mixed-scope data
- Generic empty-state handling hides still-valid completion counts
