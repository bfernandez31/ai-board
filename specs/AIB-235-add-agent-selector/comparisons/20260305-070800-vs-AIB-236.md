## Implementation Choices Analysis

### Decision Point 1: Agent Enum Iteration

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-235 | `Object.values(Agent).map()` | Yes | Extensible, auto-adapts to new agents |
| AIB-236 | Hardcoded `Agent.CLAUDE`/`Agent.CODEX` items | No | Requires manual update for new agents |

**Best**: AIB-235 - Dynamic iteration follows DRY principles and is future-proof. Used consistently across new-ticket-modal, quick-impl-modal, default-agent-card, and agent-edit-dialog.

### Decision Point 2: Quick-Impl Agent PATCH

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-235 | Reads response, updates ticket.version & ticket.agent | Yes | Prevents optimistic concurrency issues |
| AIB-236 | Fire-and-forget (no response handling) | Partial | Simpler but version gets stale |

**Best**: AIB-235 - Proper version tracking prevents 409 conflicts on subsequent operations. The transition that follows may use the stale version in AIB-236.

### Decision Point 3: Agent Badge Visibility

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-235 | Conditional: only shown when `effectiveAgent` exists | Yes | Clean, no badge noise |
| AIB-236 | Always shown, falls back to `Agent.CLAUDE` | No | Hardcoded fallback assumption |

**Best**: AIB-235 - Conditional rendering is more correct. AIB-236's fallback to `Agent.CLAUDE` is a hardcoded assumption that could be wrong if project has no defaultAgent set yet.

### Decision Point 4: Scope of Changes

| Ticket | Approach | Trade-offs |
|--------|----------|------------|
| AIB-235 | Focused on agent selector feature only | Clean, minimal diff |
| AIB-236 | Also refactors deploy icon ternary in ticket-card.tsx | Scope creep, unrelated changes |

**Best**: AIB-235 - Stays within scope. AIB-236's deploy icon refactoring is unrelated to agent selection.

### Decision Point 5: Test Coverage

| Ticket | Lines | Key Difference |
|--------|-------|----------------|
| AIB-235 | 214 | Exhaustiveness test for all enum values |
| AIB-236 | 131 | Basic render + interaction tests only |

**Best**: AIB-235 - Exhaustiveness test ensures new Agent enum values get caught if mappings are missing.

## Ranking & Recommendation

### 1. **AIB-235** - Best code (score: 91%)
**Why it wins**:
- Dynamic enum iteration (`Object.values(Agent).map()`) across all components - zero hardcoding
- Proper version tracking on quick-impl agent PATCH (reads response, updates version)
- Exhaustiveness test catches missing mappings when new agents are added
- Focused scope - no unrelated refactoring
- Conditional badge rendering avoids hardcoded `Agent.CLAUDE` fallbacks

### 2. **AIB-236** - (score: 80%)
**Why it lost**:
- Hardcoded agent items in 4 components (new-ticket-modal, quick-impl-modal, default-agent-card, agent-edit-dialog)
- Fire-and-forget PATCH in board.tsx could cause stale version conflicts
- Three hardcoded `Agent.CLAUDE` fallbacks could be wrong if defaults change
- Unrelated deploy icon refactoring in ticket-card.tsx (scope creep)
- Lighter tests (131 vs 214 lines, no exhaustiveness check)
