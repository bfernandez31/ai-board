## Implementation Choices Analysis

### [Spec] Quality Gate card with trend and distribution
| Ticket | Approach |
|--------|----------|
| AIB-374 | Embeds all data (including drawer detail) in main `/health` response. Card renders from module status props with `QualityGateCardExtras` sub-component. |
| AIB-379 | Returns summary (trend, distribution) in main `/health` response; drawer detail lazy-loaded from dedicated `/health/quality-gate` route via TanStack Query hook. Card renders extras inline. |

**Best**: AIB-379 — Hybrid approach reduces initial payload size and separates concerns. Lazy-loading aligns with TanStack Query patterns mandated by constitution.

### [Spec] Quality Gate drawer with dimensions and trend graph
| Ticket | Approach |
|--------|----------|
| AIB-374 | Data passed as props from parent `ScanDetailDrawer`. `QualityGateDrawerContent` receives pre-loaded detail. No additional fetch needed. |
| AIB-379 | `QualityGateDrawer` uses `useQualityGateDetails` TanStack Query hook with `enabled` tied to drawer open state. Loading/error states handled via query status. Recharts AreaChart for trend. |

**Best**: AIB-379 — Proper TanStack Query usage for server state (constitution III). Lazy fetch reduces wasted bandwidth when drawer is never opened. Self-contained component.

### [Spec] Last Clean card with staleness indicator
| Ticket | Approach |
|--------|----------|
| AIB-374 | `LastCleanCardExtras` sub-component in `health-module-card.tsx`. Staleness derived from `status` field. |
| AIB-379 | Inline rendering in `health-module-card.tsx` with colored left border based on `stalenessStatus`. Trend-like display with files count. |

**Best**: AIB-379 — More polished UX with visual staleness border (green/yellow/red) directly on the card.

### Decision Point 1: Data Loading Strategy
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-374 | Monolithic: all drawer data embedded in main health response | Partial — no TanStack hooks for drawer state | Simpler (one request), but larger payload on every poll; drawer data fetched even when unused |
| AIB-379 | Hybrid: summary in main response + lazy-loaded detail via dedicated routes + TanStack Query hooks | Yes — proper server state management | More files, but smaller initial payload; drawer data only fetched on open |

**Best**: AIB-379 — Constitution mandates TanStack Query for server state. Lazy loading avoids unnecessary data transfer. Dedicated routes are more RESTful.

### Decision Point 2: Type Safety in Module Cards
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-374 | Uses `as QualityGateModuleStatus` and `as LastCleanModuleStatus` type assertions in card component to narrow module types | Partial — `as` bypasses compiler checks | Simpler code, but unsafe — compiler won't catch mismatches |
| AIB-379 | Uses optional properties on `HealthModuleStatus` (`ticketCount?`, `trend?`, `distribution?`, `stalenessStatus?`) with conditional rendering | Yes — no unsafe casts | More verbose, but type-safe — compiler validates all access |

**Best**: AIB-379 — TypeScript-First (Principle I) prohibits bypassing type safety. Optional properties with conditional checks are safer than `as` casts.

### Decision Point 3: Test Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-374 | 4 test files: 2 integration (API), 2 component (drawer RTL). No unit tests for pure logic. | Partial — missing unit layer | Good coverage at integration + component layers, but pure functions untested in isolation |
| AIB-379 | 6 test files: 2 unit (pure logic), 2 integration (API), 2 component (drawer RTL). Full Testing Trophy. | Yes — all layers covered | More files, but pure functions like `computeDistribution`, `computeTrend`, `parseCleanupOutput` tested in isolation |

**Best**: AIB-379 — Constitution III mandates Testing Trophy. Pure functions (distribution, trend, staleness computation) warrant unit tests. AIB-379 follows the decision tree correctly.

### Decision Point 4: Component Organization
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-374 | Drawers as content components (`*-drawer-content.tsx`) rendered inside existing `ScanDetailDrawer` shell | Yes — feature folders | Reuses existing drawer shell, but couples new modules to scan drawer infrastructure |
| AIB-379 | Drawers as standalone Sheet components (`*-drawer.tsx`) with independent open/close state, rendered from `health-dashboard.tsx` | Yes — feature folders | More self-contained, cleaner separation from scan-based drawers |

**Best**: AIB-379 — Quality Gate and Last Clean are not scan-based modules. Standalone drawers avoid coupling to the scan drawer abstraction.
