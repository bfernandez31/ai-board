## Implementation Choices Analysis

### Decision Point 1: Agent Schema Location
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-228 | Dedicated `agent.ts` schema file | Yes (separation of concerns) | Cleaner imports, standalone |
| AIB-233 | Added to `clarification-policy.ts` | Acceptable (colocation) | Less files, but mixed concerns |

**Best**: AIB-228 — Separate file follows single-responsibility, consistent with how `ClarificationPolicy` has its own file.

### Decision Point 2: Stage Restriction for Agent Updates
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-228 | Agent restricted to INBOX stage (with description/policy) | Yes (security-first) | Prevents mid-workflow agent changes |
| AIB-233 | No stage restriction for agent | Risk | Agent could change mid-workflow |

**Best**: AIB-228 — Restricting agent updates to INBOX is defensive and correct. Changing agent mid-workflow could cause inconsistent behavior.

### Decision Point 3: Project defaultAgent in Ticket GET Response
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-228 | Returns `defaultAgent` in ticket's project object | Yes | Client can resolve effective agent |
| AIB-233 | Does not include `defaultAgent` in GET response | Gap | Client cannot resolve agent inheritance |

**Best**: AIB-228 — Client needs project `defaultAgent` to display effective agent when `ticket.agent` is null.

### Decision Point 4: Multipart Form Data Handling
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-228 | Parses `agent` from multipart fields | Complete | Full API coverage |
| AIB-233 | No multipart agent handling | Incomplete | File upload + agent won't work |

**Best**: AIB-228 — Existing ticket creation uses multipart for image attachments. Agent field must be handled there too.

### Decision Point 5: Code Refactoring (Beyond Feature)
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-228 | Extracted `isWorkflowTestMode`, `formatDurationMs`, consolidated `TicketWithVersion` | Yes (DRY) | Cleaner codebase, slight scope creep |
| AIB-233 | No unrelated changes | Focused | Missed DRY opportunities |

**Best**: AIB-228 — DRY improvements reduce duplication across 3 workflow dispatchers. The refactoring is clean and well-scoped.

### Decision Point 6: Agent Resolution Utility
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-228 | Created `resolveEffectiveAgent()` utility | Yes | Reusable resolution logic |
| AIB-233 | No resolution utility | Simpler | Resolution logic must be duplicated |

**Best**: AIB-228 — Centralizes `ticketAgent ?? projectDefaultAgent` logic for consistent behavior.
