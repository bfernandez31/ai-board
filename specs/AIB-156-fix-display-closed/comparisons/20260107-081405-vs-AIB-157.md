## Implementation Choices Analysis

### Decision Point 1: Data Fetching Architecture

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-156 | Dedicated `useTicketByKey` hook with TanStack Query | ✅ Principle I | Reusable, cached, follows existing patterns |
| AIB-157 | Inline `fetch` in useEffect | ❌ Bypasses TanStack Query | One-off solution, no caching, no error states |

**Best**: AIB-156 - Uses TanStack Query per constitution state management standards. Provides automatic caching, refetch on focus, and error handling. The hook is reusable across components.

**Code Evidence**:
```typescript
// AIB-156: Proper TanStack Query hook
export function useTicketByKey(projectId, ticketKey, enabled) {
  return useQuery({
    queryKey: queryKeys.projects.ticketByKey(projectId, ticketKey),
    queryFn: async () => { /* ... */ },
    enabled: enabled && !!ticketKey,
    staleTime: 5000,
    gcTime: 10 * 60 * 1000,
  });
}

// AIB-157: Inline fetch
const fetchTicketAndOpen = async () => {
  const response = await fetch(`/api/projects/${projectId}/tickets/${ticketKey}`);
  // ...
};
```

### Decision Point 2: Query Key Management

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-156 | Added `ticketByKey` to centralized query keys | ✅ Principle V | Consistent cache invalidation |
| AIB-157 | Manual `setQueryData` in component | ⚠️ Direct cache mutation | Bypass query system, potential stale data |

**Best**: AIB-156 - Extends the centralized `query-keys.ts` file with a new key pattern. This allows proper cache invalidation across the app.

**Code Evidence**:
```typescript
// AIB-156: Centralized query key
ticketByKey: (projectId: number, ticketKey: string) =>
  ['projects', projectId, 'tickets', 'by-key', ticketKey] as const,

// AIB-157: Manual cache mutation
queryClient.setQueryData(
  queryKeys.projects.tickets(projectId),
  (old: TicketWithVersion[] = []) => [...old, fetchedTicket]
);
```

### Decision Point 3: State Management in Component

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-156 | `pendingTicketKey` state + 2 useEffects | ✅ Separation of concerns | Clear control flow, testable |
| AIB-157 | Single useEffect with async fetch | ⚠️ Side effects in render path | Harder to test, mixed concerns |

**Best**: AIB-156 - Separates the "trigger fetch" logic from the "handle result" logic. First effect sets `pendingTicketKey`, second effect responds to `fetchedTicket`. This is more testable and follows React best practices.

## Ranking & Recommendation

### 1. **AIB-156** - Best code (score: 88%)
**Why it wins**:
- **Dedicated hook abstraction**: `useTicketByKey` follows existing patterns (`useTicket`, `useTickets`) and is reusable
- **TanStack Query compliance**: Uses proper query caching, stale-while-revalidate, and automatic error handling per constitution
- **Comprehensive integration tests**: 163 lines testing API behavior, closed tickets, 404 handling, field validation
- **Centralized query keys**: Extends `query-keys.ts` for consistent cache invalidation
- **Separation of concerns**: Two-step useEffect pattern (trigger → handle) is more testable

**Architecture**:
```
useTicketByKey hook (TanStack Query)
  ↓
queryKeys.projects.ticketByKey (centralized)
  ↓
Existing GET /api/projects/:id/tickets/:key endpoint
```

### 2. **AIB-157** - (score: 72%)
**Why it lost**:
- **Inline fetch**: Bypasses TanStack Query, no automatic caching or error states
- **Manual cache mutation**: Uses `setQueryData` to inject ticket into cache, bypassing query system
- **Limited tests**: Only component test, no API verification
- **Single useEffect**: Mixes async fetch, cache mutation, and error handling in one effect

**Architecture**:
```
useEffect with inline fetch
  ↓
Manual queryClient.setQueryData
  ↓
Existing GET /api/projects/:id/tickets/:key endpoint
```
