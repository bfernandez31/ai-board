## Implementation Choices Analysis

### Decision Point 1: Credentials provider design
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-313 | Centralized `authorizeDevLogin` with Zod parsing, preview-only enablement, explicit `DEV_LOGIN_ENABLED`, timing-safe secret compare | Yes | Most robust and easiest to audit |
| AIB-314 | Shared auth helper with Zod and timing-safe compare, but enabled in local dev when a secret exists | Mostly | Good structure, looser enablement boundary |
| AIB-316 | Inline provider `authorize`, direct secret equality, enabled whenever `DEV_LOGIN_SECRET` exists | Warning | Simpler, but weaker security posture |
| AIB-315 | Same as AIB-316 without error-path hardening | Warning | Smallest patch, least complete |
| AIB-311 | Inline provider plus deterministic user ids | Warning | Stable ids, but weak runtime validation |

**Best**: AIB-313. It is the only implementation that combines explicit environment gating, schema validation, and secret comparison hardening in one reusable path.

### Decision Point 2: Sign-in UX and error handling
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-313 | Server action + callback interception + redirect back to sign-in with a specific dev-login error | Yes | Best user-facing failure path |
| AIB-314 | Server action with inline form and query-string error rendering | Mostly | Clean UI, but no callback interception layer |
| AIB-316 | Client component calls `next-auth/react` and manages redirect/error in browser | Warning | More client code and duplicated auth flow |
| AIB-315 | Same client-side approach with less test coverage | Warning | Lightweight, but more brittle |
| AIB-311 | Same client-side approach and env mismatch for form visibility | No | Can hide/show UI inconsistently |

**Best**: AIB-313. It keeps auth flow server-first and fixes the default credentials failure redirect so the UX stays on the sign-in page.

### Decision Point 3: Test strategy
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-313 | Unit + integration + e2e + page tests | Yes | Highest confidence |
| AIB-314 | Focused unit tests around auth helper, page, and user service | Mostly | Good coverage, less end-to-end confidence |
| AIB-316 | Unit tests for form and user service | Partial | UI covered, auth path not deeply verified |
| AIB-311 | Unit tests include several assertion-only pseudo-tests | Warning | Coverage count overstates confidence |
| AIB-315 | User-service tests only | No | Misses UI and login flow behavior |

**Best**: AIB-313. It is the only variant that verifies the actual login behavior across layers.

## Ranking & Recommendation
