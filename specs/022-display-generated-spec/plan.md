## Summary

**Feature**: Add functionality to retrieve and display spec.md files generated by GitHub Actions workflows after "specify" command completion.

**Primary Requirement**: When a user clicks on a ticket that has a completed "specify" job, show a "View Specification" button that opens a modal dialog displaying the markdown specification document.

**Technical Approach** (from research.md):
- New API route: `GET /api/projects/:projectId/tickets/:id/spec`
- GitHub API integration via Octokit (existing dependency)
- React component: SpecViewer using shadcn/ui Dialog
- Markdown rendering: react-markdown + react-syntax-highlighter
- Test mode detection for E2E tests (mock GitHub API)
- Project-scoped validation following existing patterns

## Technical Context

**Language/Version**: TypeScript 5.6 (strict mode), Node.js 22.20.0 LTS

**Primary Dependencies**:
- **Existing**: Next.js 15, React 18, Prisma 6.x, @octokit/rest, shadcn/ui, Zod 4.x, Playwright
- **New**: react-markdown ^9.0.1, react-syntax-highlighter ^15.5.0

**Storage**: PostgreSQL 14+ via Prisma ORM (no schema changes)

**Testing**: Playwright E2E tests with GitHub API mocking

**Target Platform**: Web (Next.js App Router, Vercel deployment)

**Project Type**: web (Next.js monorepo with app/, components/, lib/, tests/)

**Performance Goals**:
- API response: <200ms (test mode <50ms)
- Markdown render: <100ms
- Modal open: <200ms Time to Interactive
- Bundle impact: ~50KB for spec-viewer chunk

**Constraints**:
- Must work in test mode without external API calls
- Must handle large markdown files (>1MB) gracefully
- Must sanitize markdown to prevent XSS (react-markdown default)
- Must follow existing API route patterns (project-scoped validation)

**Scale/Scope**:
- Single feature (view-only, no editing)
- 1 new API route
- 1 new React component (SpecViewer)
- 1 modified component (TicketDetailModal - add button)
- ~15 functional requirements
- Estimated: 30-35 implementation tasks

## Phase 2: Task Planning Approach

*This section describes what the /tasks command will do - DO NOT execute during /plan*

### Task Generation Strategy

The /tasks command will:
1. Load `.specify/templates/tasks-template.md` as base structure
2. Generate tasks from Phase 1 artifacts (contracts, data model, quickstart)
3. Apply TDD ordering: Tests before implementation
4. Apply dependency ordering: Backend → Frontend → Integration
5. Mark parallelizable tasks with [P] flag

**Category 1: Setup & Dependencies** (2 tasks)
- Install new npm packages (react-markdown, react-syntax-highlighter)
- Create directory structure and placeholder files

**Category 2: Backend - API Route** (6-8 tasks)
1. Create validation schemas (Zod)
2. Create GitHub API integration module
3. Create API route handler (GET /api/projects/:projectId/tickets/:id/spec)
4. Implement project/ticket validation logic
5. Implement job status checking logic
6. Implement GitHub API call with test mode detection
7. Implement error handling and response formatting
8. Add logging for debugging

**Category 3: Frontend - Components** (8-10 tasks)
1. Create SpecViewer component (modal shell)
2. Add Dialog setup (shadcn/ui)
3. Implement markdown rendering (react-markdown)
4. Add syntax highlighting (react-syntax-highlighter)
5. Implement loading state
6. Implement error state
7. Add modal close handlers (button + ESC)
8. Style component for dark theme
9. Modify TicketDetailModal (add button + visibility logic)
10. Implement button click handler and state management

**Category 4: Testing** (12-15 tasks)
1. Create test fixtures (mock spec content)
2. E2E test: Button visibility - has branch + completed job [must fail initially]
3. E2E test: Button visibility - no branch [must fail initially]
4. E2E test: Button visibility - no completed job [must fail initially]
5. E2E test: Button visibility - multiple jobs [must fail initially]
6. E2E test: Spec content display [must fail initially]
7. E2E test: Markdown rendering (headings, lists, code) [must fail initially]
8. E2E test: Syntax highlighting [must fail initially]
9. E2E test: Loading state [must fail initially]
10. E2E test: Error handling - 404 [must fail initially]
11. E2E test: Error handling - 403 [must fail initially]
12. E2E test: Modal close (button) [must fail initially]
13. E2E test: Modal close (ESC) [must fail initially]
14. E2E test: Scrollable content [must fail initially]
15. E2E test: Responsive layout [must fail initially]

**Category 5: Integration & Refinement** (5-7 tasks)
1. Run all E2E tests (verify they pass)
2. Performance testing (API response time, render time)
3. Accessibility testing (keyboard navigation, screen readers)
4. Cross-browser testing (Chrome, Firefox, Safari)
5. Mobile testing (responsive design)
6. Code review checklist
7. Documentation updates (README, API docs)

### Task Ordering Strategy

**TDD Order**:
1. Write failing tests first (Category 4)
2. Implement backend to make tests pass (Category 2)
3. Implement frontend to make tests pass (Category 3)
4. Validate and refine (Category 5)

**Dependency Order**:
```
Setup (Category 1)
  ↓
Backend API Route (Category 2) ← Tests written first (Category 4)
  ↓
Frontend Components (Category 3) ← Tests written first (Category 4)
  ↓
Integration & Validation (Category 5)
```

**Parallel Execution** [P]:
- Setup tasks can run in parallel
- Backend and Frontend can have parallel subtasks within category
- Tests can be written in parallel
- Final validation steps run sequentially
