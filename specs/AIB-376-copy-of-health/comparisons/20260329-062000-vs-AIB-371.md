## Implementation Choices Analysis

### [Spec] Issues grouped by module type (FR-004)
| Ticket | Approach |
|--------|----------|
| AIB-376 | Renders raw markdown report via `ScanReportContent` (generic markdown renderer). No module-specific grouping. |
| AIB-371 | `DrawerIssues` component with discriminated union switch rendering per module type (Security by severity, Compliance by category, Tests auto-fixed/non-fixable, etc.) |

**Best**: AIB-371 - Fully implements module-specific issue grouping per spec with typed renderers.

### [Spec] Ticket links navigate to board (FR-007)
| Ticket | Approach |
|--------|----------|
| AIB-376 | `GeneratedTicketsSection` shows ticket keys but uses `<div>` (no navigation link) |
| AIB-371 | `DrawerTickets` uses `<a href>` with proper board URL (`/projects/{id}/board?ticket={key}`) |

**Best**: AIB-371 - Implements actual navigation per spec requirement.

### Decision Point 1: Report Data Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-376 | New `/scans/latest` and `/scans/[scanId]` API routes; renders report as raw markdown string | IV. Yes (Zod on latest route) | Simple but no structured issue display |
| AIB-371 | Extends existing `/scans` route with `includeReport` param; Zod schema validation in `lib/health/report-schemas.ts`; typed discriminated union `ScanReport` | IV. Yes (Zod schemas) | Richer data layer, reusable parsing, DRY API |

**Best**: AIB-371 - Extends existing API (DRY), adds typed report parsing with Zod validation, enables module-specific rendering.

### Decision Point 2: Component Organization
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-376 | Flat structure in `components/health/` with `shared.ts` for shared constants; monolithic `scan-detail-drawer.tsx` (167 lines) | II. Partial | Simpler file tree but less modular |
| AIB-371 | `components/health/drawer/` sub-folder with dedicated files (header, history, issues, states, tickets); `lib/health/types.ts` extended with report types | II. Yes | Better separation of concerns, each file focused |

**Best**: AIB-371 - Feature-based folder structure per constitution Principle II.

### Decision Point 3: State Management
| Ticket | Approach | Constitution |
|--------|----------|--------------|
| AIB-376 | TanStack Query via dedicated hooks (`useScanReport`, `useScanHistory`, `useGeneratedTickets`) | Yes |
| AIB-371 | TanStack Query via `useScanReport` hook + inline `useQuery` in `DrawerHistory` | Yes |

**Best**: AIB-376 - Slightly cleaner with all hooks centralized, but both are valid.
