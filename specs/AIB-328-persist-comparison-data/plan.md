# Implementation Plan: Persist Comparison Data to Database via Workflow

**Branch**: `AIB-328-persist-comparison-data` | **Date**: 2026-03-21 | **Spec**: `specs/AIB-328-persist-comparison-data/spec.md`
**Input**: Feature specification from `/specs/AIB-328-persist-comparison-data/spec.md`

## Summary

Bridge the `/compare` command's markdown output with the database-backed comparison dashboard by: (1) adding JSON data file generation to the compare command, (2) creating a POST endpoint for comparison persistence, and (3) adding a workflow step that reads the JSON file and POSTs it to the API. All three changes are additive — the existing markdown report workflow is unaffected by failures in any new component.

## Implementation Details

### Change 1: Compare Command — JSON Data File (FR-001, FR-002, FR-003, FR-004)

**File**: `.claude/commands/ai-board.compare.md`

Add a new step after Step 10 (markdown report generation) and before Step 11 (result file):

**Step 10.5: Write Comparison Data JSON**

The command already builds the `ComparisonReport` object and has access to ticket metadata (source ticket, participants with IDs, stages, workflow types, agents). After writing the markdown report, construct and write a JSON file containing:

```typescript
{
  projectId: number,           // From TICKET_KEY → project lookup
  sourceTicket: {              // Source ticket metadata
    id, ticketKey, title, stage, workflowType, agent
  },
  participants: [{             // Each compared ticket
    id, ticketKey, title, stage, workflowType, agent
  }],
  markdownPath: string,        // Path to the markdown report just written
  report: ComparisonReport     // The full report object
}
```

**File naming**: Same as markdown but `.json` extension: `{timestamp}-vs-{keys}.json`
**Location**: Same directory: `specs/{branch}/comparisons/`
**Failure handling**: Wrap in try-catch. If JSON write fails, log warning and continue to Step 11.

### Change 2: POST Endpoint (FR-008, FR-009, FR-010, FR-011)

**File**: `app/api/projects/[projectId]/comparisons/route.ts`

Add a `POST` export function:

1. **Auth**: `validateWorkflowAuth(request)` → 401 on failure
2. **Parse**: `request.json()` → parse body
3. **Validate**: Zod schema validating the `PersistComparisonInput` structure:
   - `projectId` must be a positive integer matching the URL path parameter
   - `sourceTicket` and `participants` must have required ticket fields
   - `markdownPath` must be a non-empty string
   - `report` must have required top-level fields
4. **Persist**: Call `persistComparisonRecord()` from `lib/comparison/comparison-record.ts`
5. **Response**: `201 Created` with `{ id: record.id, generatedAt: record.generatedAt }`
6. **Errors**:
   - 400 for validation failures (with Zod error details)
   - 400 for Prisma foreign key constraint violations (referenced tickets don't exist)
   - 500 for unexpected errors

**Zod schema** (new, defined in the route file or a shared validator):

```typescript
const persistComparisonSchema = z.object({
  projectId: z.number().int().positive(),
  sourceTicket: z.object({
    id: z.number().int().positive(),
    ticketKey: z.string().min(1),
    title: z.string().min(1),
    stage: z.enum(['INBOX', 'SPECIFY', 'PLAN', 'BUILD', 'VERIFY', 'SHIP', 'CLOSED']),
    workflowType: z.enum(['FULL', 'QUICK', 'CLEAN']),
    agent: z.enum(['CLAUDE', 'CODEX']).nullable(),
  }),
  participants: z.array(z.object({
    id: z.number().int().positive(),
    ticketKey: z.string().min(1),
    title: z.string().min(1),
    stage: z.enum(['INBOX', 'SPECIFY', 'PLAN', 'BUILD', 'VERIFY', 'SHIP', 'CLOSED']),
    workflowType: z.enum(['FULL', 'QUICK', 'CLEAN']),
    agent: z.enum(['CLAUDE', 'CODEX']).nullable(),
  })),
  markdownPath: z.string().min(1),
  report: z.object({
    metadata: z.object({
      generatedAt: z.string().or(z.date()),
      sourceTicket: z.string(),
      comparedTickets: z.array(z.string()).min(1),
      filePath: z.string(),
    }),
    summary: z.string(),
    alignment: z.object({}).passthrough(),
    implementation: z.record(z.object({}).passthrough()),
    compliance: z.record(z.object({}).passthrough()),
    telemetry: z.record(z.object({}).passthrough()),
    recommendation: z.string(),
    warnings: z.array(z.string()),
  }),
});
```

### Change 3: Workflow Persistence Step (FR-005, FR-006, FR-007, FR-012)

**File**: `.github/workflows/ai-board-assist.yml`

Add a new step after the Claude command execution block and before the "Commit and Push Changes" step. Only runs for `/compare` commands:

```yaml
- name: Persist Comparison Data
  if: ${{ env.SKIP_CLAUDE != 'true' && contains(inputs.comment, '/compare') }}
  working-directory: target
  env:
    APP_URL: ${{ vars.APP_URL }}
    WORKFLOW_API_TOKEN: ${{ secrets.WORKFLOW_API_TOKEN }}
    PROJECT_ID: ${{ inputs.project_id }}
    BRANCH: ${{ inputs.branch }}
  run: |
    echo "📊 Checking for comparison data JSON..."
    COMPARISONS_DIR="specs/$BRANCH/comparisons"

    if [ ! -d "$COMPARISONS_DIR" ]; then
      echo "⚠️ No comparisons directory found, skipping persistence"
      exit 0
    fi

    # Find the newest JSON file
    JSON_FILE=$(ls -t "$COMPARISONS_DIR"/*.json 2>/dev/null | head -1)

    if [ -z "$JSON_FILE" ]; then
      echo "⚠️ No comparison JSON file found, skipping persistence"
      exit 0
    fi

    echo "📄 Found comparison data: $JSON_FILE"

    # POST to persistence endpoint
    HTTP_RESPONSE=$(curl -s -w "\n%{http_code}" -X POST \
      "${APP_URL}/api/projects/${PROJECT_ID}/comparisons" \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer ${WORKFLOW_API_TOKEN}" \
      -d @"$JSON_FILE")

    HTTP_CODE=$(echo "$HTTP_RESPONSE" | tail -n1)
    RESPONSE_BODY=$(echo "$HTTP_RESPONSE" | sed '$d')

    if [ "$HTTP_CODE" = "201" ]; then
      echo "✅ Comparison data persisted successfully"
      echo "   Response: $RESPONSE_BODY"
    else
      echo "⚠️ Comparison persistence failed (HTTP $HTTP_CODE)"
      echo "   Response: $RESPONSE_BODY"
      echo "   Continuing workflow — markdown report is the primary artifact"
    fi
```

**Key design decisions**:
- `exit 0` on all paths — never fails the workflow
- Uses `curl -d @file` to send JSON file contents directly
- Logs outcome for observability (FR-012)
- Runs only for `/compare` commands via the `if` condition

## Testing Strategy

### Integration Tests (Primary)

**File**: `tests/integration/comparisons/comparison-persistence-endpoint.test.ts`

| Test Case | Type | What It Verifies |
|-----------|------|------------------|
| POST with valid payload creates comparison record | Integration | FR-008, FR-010, SC-005 |
| POST returns 201 with record ID and timestamp | Integration | FR-010 |
| POST with missing auth returns 401 | Integration | FR-009 |
| POST with invalid token returns 401 | Integration | FR-009 |
| POST with invalid payload returns 400 | Integration | FR-011 |
| POST with non-existent ticket IDs returns 400 | Integration | FR-011, edge case |
| POST with projectId mismatch returns 400 | Integration | FR-011 |
| POST creates all related entities (participants, metrics, compliance, decisions) | Integration | SC-005, SC-006 |
| Multiple POSTs for same tickets create separate records | Integration | Edge case (point-in-time snapshots) |

**Test approach**: Use Vitest with real Prisma client against test database. Create test tickets via Prisma, then POST comparison data referencing those tickets.

### Not Tested (by design)

- **E2E tests**: Not needed — no browser interaction. The endpoint is called by a workflow, not a user.
- **Command JSON output**: The compare command runs inside Claude — testing JSON file output is not feasible in automated tests. Validated by the strict Zod schema at the API boundary.
- **Workflow step**: Tested implicitly by running the actual workflow. Shell script logic is straightforward (find file, curl POST).
