## Implementation Choices Analysis

### [Spec] Command Palette Implementation
| Ticket | Approach |
|--------|----------|
| AIB-356 | Uses shadcn/ui `Command` component (built on `cmdk` library) — standard shadcn pattern |
| AIB-357 | Builds custom command palette from `Dialog` + `Input` + custom results component (112 lines) |

**Best**: AIB-356 — Uses the canonical shadcn Command component, which is the constitution-mandated approach ("Use shadcn/ui components exclusively for UI primitives"). AIB-357's from-scratch implementation adds unnecessary complexity and violates "No custom component styling from scratch—compose shadcn/ui components instead."

### [Spec] Keyboard Shortcuts Coexistence
| Ticket | Approach |
|--------|----------|
| AIB-356 | Dispatches `CustomEvent('open-command-palette')` from board; palette listens via `useEffect` |
| AIB-357 | Adds `blocked` prop to `useKeyboardShortcuts` hook; palette dispatches state change events |

**Best**: AIB-357 — The `blocked` prop is a cleaner API that makes shortcut suppression explicit in the hook contract rather than relying on DOM events for state synchronization.

### Decision Point 1: Layout Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | `app/projects/[projectId]/layout.tsx` — Next.js nested layout with CSS Grid | Yes — App Router convention | Single layout file, automatic nesting |
| AIB-357 | `ProjectShell` wrapper component — manually wraps each page (4 files modified) | Acceptable | Requires wrapping every project page |

**Best**: AIB-356 — Uses idiomatic Next.js App Router nested layout pattern. Single file controls rail + content layout for all project routes. AIB-357's approach requires wrapping each page manually and is less maintainable as pages are added.

### Decision Point 2: Search Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | Reuses existing `useTicketSearch` hook (TanStack Query) + client-side nav filtering | Yes | Minimal new code, leverages existing infra |
| AIB-357 | New API route `/command-palette` + 201-line ranking algorithm + Zod schema + TanStack Query hook | Yes (Zod validation) | Over-engineered for 4 nav items + ticket search |

**Best**: AIB-356 — Reuses existing infrastructure. AIB-357 creates a dedicated API endpoint with a 201-line ranking algorithm and new Zod schema for what is essentially fuzzy matching over 4 navigation items and existing ticket search. This violates "Don't create helpers, utilities, or abstractions for one-time operations" and "the minimum needed for the current task."

### Decision Point 3: Navigation Data Model
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | `nav-items.ts` (17 lines) — flat array with id, label, icon, href, group | Yes | Simple, direct |
| AIB-357 | `project-destinations.ts` (95 lines) — definitions with keywords, descriptions, icon keys, builder functions | Yes | Over-abstracted for 4 items |

**Best**: AIB-356 — 17 lines vs 95 lines for the same 4 navigation items. AIB-357 adds keyword arrays, description fields, icon key mappings, and multiple utility functions that add complexity without proportional benefit.

### Decision Point 4: State Management for Palette
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | `useState` in layout + prop drilling to CommandPalette; `useTicketSearch` for server state | Partial | TanStack Query for tickets, useState for UI |
| AIB-357 | `useState` in header + `useCommandPalette` (TanStack Query) for all palette data | Yes | Proper server state via TanQuery |

**Best**: Tie — Both use TanStack Query for server state. AIB-357 has a dedicated hook but it queries a new API that wasn't needed.
