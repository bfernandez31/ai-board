## Implementation Choices Analysis

### [Spec] Fuzzy matching across navigation + tickets
| Ticket | Approach |
|--------|----------|
| AIB-356 | Client-side fuzzy match via cmdk built-in filtering; ticket search reuses existing `useTicketSearch` hook with simple string matching |
| AIB-357 | Dedicated server-side `/api/projects/:id/command-palette` endpoint with Zod-validated query params, `command-palette-ranking.ts` module with explicit scoring algorithm |

**Best**: AIB-357 - Purpose-built ranking engine with explicit scoring; server-side search allows DB-level optimization. AIB-356 relies on cmdk's built-in filter which is less controllable.

### [Spec] Header simplification (remove nav icons)
| Ticket | Approach |
|--------|----------|
| AIB-356 | Fully removes Specs, Analytics, Activity links from header; removes `githubOwner`/`githubRepo` props from MobileMenu |
| AIB-357 | Keeps Specs (FileText) icon in header; removes Analytics and Activity only |

**Best**: AIB-356 - More faithfully implements the spec ("remove the icon buttons from the header"). AIB-357 retains the Specs link which contradicts the acceptance criteria.

### Decision Point 1: Architecture / Layout Strategy
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | New `app/projects/[projectId]/layout.tsx` as client component wrapping sidebar + command palette | Partial (makes layout client component) | Clean layout-level integration; but forces entire layout tree into client |
| AIB-357 | `ProjectShell` server component wrapper, each page explicitly wraps content | Yes (Server Components by default) | More explicit, keeps server component defaults; slightly more boilerplate per page |

**Best**: AIB-357 - `ProjectShell` is a server component composing `DesktopProjectRail` (client). Follows constitution principle II (Server Components by default). AIB-356's layout.tsx is `'use client'` which forces the entire subtree client-side.

### Decision Point 2: Command Palette Implementation
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | Uses `cmdk` library (shadcn Command component); palette lives in layout | Yes (shadcn/ui) | Well-tested library; less custom code; but adds dependency |
| AIB-357 | Custom Dialog + Input + results component; dedicated API route + TanStack Query hook | Yes (TanStack Query for server state) | More code but full control; proper server-state management |

**Best**: AIB-357 - Uses TanStack Query for server-state (constitution compliance), dedicated API with Zod validation (constitution IV: Security-First), and proper type definitions. AIB-356 uses cmdk which is valid (Radix-based) but ticket search still goes through the old `useTicketSearch` hook rather than a unified search approach.

### Decision Point 3: Navigation Data Model
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | Simple `nav-items.ts` array with `NavigationItem` interface (17 lines) | Yes | Minimal, easy to understand |
| AIB-357 | `project-destinations.ts` with `DestinationDefinition`, helper functions for href building, pathname matching, icons export (95 lines); types in `lib/types.ts` | Yes | More structured; reusable across rail + palette + ranking |

**Best**: AIB-357 - Types defined in `lib/types.ts` (proper shared type location), helper functions enable reuse across rail, palette, and ranking. AIB-356's approach is simpler but duplicates path logic across components.

### Decision Point 4: State Communication (palette <-> board shortcuts)
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-356 | `CustomEvent('open-command-palette')` from board; palette state managed in layout | Acceptable | Simple; but one-directional (board can open palette, can't know if it's open) |
| AIB-357 | Bidirectional CustomEvents: `project-command-palette:open` + `project-command-palette:state-change`; board tracks palette open state to block shortcuts | Better | Board knows palette state; properly blocks shortcuts when palette is open |

**Best**: AIB-357 - Bidirectional event system enables the board to block keyboard shortcuts while the palette is open, directly implementing FR-015.
