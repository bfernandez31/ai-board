## Implementation Choices Analysis

### [Spec] Winner score badge display
| Ticket | Approach |
|--------|----------|
| AIB-365 | Backend change in `project-comparison-summary.ts` to expose `winnerScore`; renders Trophy icon + score badge |
| AIB-366 | No backend change; Trophy icon shown but no numeric score displayed |

**Best**: AIB-365 - Full-stack implementation delivers the spec requirement for score display.

### [Spec] Deep link auto-fetch for out-of-page comparisons
| Ticket | Approach |
|--------|----------|
| AIB-365 | `deepLinkAttemptedRef` + useEffect loop that fetches next pages until target comparison is found |
| AIB-366 | Sets `expandedId` from URL param but only scrolls if comparison is already loaded; no auto-fetch |

**Best**: AIB-365 - Handles the spec scenario where deep-linked comparison is not in initial page.

### Decision Point 1: Accordion Component Pattern
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-365 | Radix `Collapsible` from shadcn/ui with `CollapsibleContent` for animated expand | Yes (Principle II) | Accessible, built-in animation, proper ARIA |
| AIB-366 | Conditional rendering `{isExpanded && <div>}` | Partial | No animation, simpler code, missing accessibility |

**Best**: AIB-365 - Uses shadcn/ui primitives per constitution Principle II; provides animation and ARIA attributes.

### Decision Point 2: Component Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-365 | Separate `comparison-card.tsx` file with typed props interface in `types.ts` | Yes (Principle II) | Feature-folder pattern, testable in isolation |
| AIB-366 | Inline `ComparisonCard` + `ExpandedDetail` functions in `project-comparisons-page.tsx` | Partial | Simpler file structure but harder to test individually |

**Best**: AIB-365 - Feature-folder component separation aligns with constitution; enables focused unit testing.

### Decision Point 3: Detail Data Fetching
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-365 | Single `detailQuery` at page level, passed as prop to expanded card | Yes | One query, clean data flow, parent controls loading state |
| AIB-366 | `ExpandedDetail` component with own `useProjectComparisonDetail` hook | Partial | Component-local data, but creates/destroys query on expand/collapse |

**Best**: AIB-365 - Avoids query churn on expand/collapse; parent-managed state is more predictable.

### Decision Point 4: Infinite Query Hook Design
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-365 | Uses `select` option to flatten pages into `comparisons` array at query level | Yes | Derived state computed by TanStack Query, memoized automatically |
| AIB-366 | Flattens pages with `useMemo` + `flatMap` in the component | OK | Works but duplicates what `select` provides natively |

**Best**: AIB-365 - Leverages TanStack Query's `select` for cleaner derived state.
