## Implementation Choices Analysis

### [Spec] Scan History Pagination
| Ticket | Approach |
|--------|----------|
| AIB-375 | Page-based pagination (page + pageSize) |
| AIB-370 | Cursor-based pagination (cursor + limit + hasMore) |

**Best**: AIB-370 — Cursor-based is more scalable and consistent with modern API patterns for ordered datasets.

### [Spec] Aurora Theme Compliance
| Ticket | Approach |
|--------|----------|
| AIB-375 | Uses `aurora-bg-subtle` on some cards, but GlobalScoreCard uses `border-ctp-mauve/15` directly |
| AIB-370 | Consistently uses `aurora-bg-section`, `aurora-glass`, `aurora-glass-hover`, `aurora-glow-score` |

**Best**: AIB-370 — Full aurora theme adoption matching CLAUDE.md requirements.

### Decision Point 1: State Management for Scan Triggering
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-375 | Raw `useState` + `fetch` in `ScanActionButton` | No — violates TanStack Query requirement | Simpler but no cache invalidation, no error handling via query |
| AIB-370 | `useTriggerScan` mutation hook via TanStack Query | Yes — proper mutation with `onSuccess` cache invalidation | Follows constitution, automatic cache sync |

**Best**: AIB-370 — Constitution mandates TanStack Query for server state mutations.

### Decision Point 2: Query Key Organization
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-375 | Inline query keys `['projects', projectId, 'health']` | Acceptable but fragile | Risk of key drift |
| AIB-370 | Centralized `queryKeys.health.*` in `query-keys.ts` | Yes — follows existing pattern | Consistent, refactor-safe |

**Best**: AIB-370 — Extends existing centralized query key system.

### Decision Point 3: Type System Design
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-375 | Array-based module responses, `ModuleConfig` with LucideIcon in types.ts | Yes | Clean but mixes UI concerns into shared types |
| AIB-370 | Object-keyed modules in response, separate `HealthModuleType` union, reuses `lib/quality-score` | Yes | Better separation, reuses existing score utilities |

**Best**: AIB-370 — Reuses existing `getScoreColor`/`getScoreThreshold` from `lib/quality-score` instead of re-exporting wrappers, and provides richer type definitions.

### Decision Point 4: API Response Structure
| Ticket | Approach |
|--------|----------|
| AIB-375 | Array of modules, separate `queries.ts` with `computePassiveModuleScores`, `getLatestScans` |
| AIB-370 | Object-keyed modules, inline queries in route handler, includes `activeScans` array + `color` config |

**Best**: AIB-370 — Returns `activeScans` in response enabling client to detect scanning state without iterating modules. Includes `color` config in response reducing client computation.

### Decision Point 5: Scan Status Callback Robustness
| Ticket | Approach |
|--------|----------|
| AIB-375 | Validates transitions but no idempotency handling |
| AIB-370 | Idempotent status updates (same status returns current state), PENDING→FAILED allowed |

**Best**: AIB-370 — More robust for workflow retries and error scenarios.

## Constitution Compliance (Code Only)

**AIB-375 issues**: Uses raw `useState` + `fetch` for scan triggering instead of TanStack Query mutation (Constitution: "Server state: TanStack Query v5 for data fetching, caching, and mutations"). `queries.ts` helper module is well-structured but re-exports score utilities redundantly.

**AIB-370 strengths**: Proper `useTriggerScan` mutation hook, centralized query keys, reuses existing `lib/quality-score`, fuller aurora theme adoption, cursor-based pagination, idempotent status callbacks.
