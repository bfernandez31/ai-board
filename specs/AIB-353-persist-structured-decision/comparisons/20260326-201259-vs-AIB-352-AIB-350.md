## Implementation Choices Analysis

### Decision Point 1: Decision Point Data Structure
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-353 | Array with `ticketKey`+`summary` objects, field named `approaches` | Yes | Clean, ordered |
| AIB-352 | Array with `ticketKey`+`summary` objects, field named `participantApproaches` (matches DB) | Yes | Best — aligns with DB schema naming |
| AIB-350 | `Record<string, string>` object map | Partial | Loses insertion order guarantee, no ticketId mapping |

**Best**: AIB-352 — Uses array (preserves order) with naming that matches the Prisma schema (`participantApproaches`), avoiding any impedance mismatch.

### Decision Point 2: Backwards Compatibility / Fallback
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-353 | Structured path with fallback to `matchingRequirements` derivation | Yes | Safe, preserves legacy behavior |
| AIB-352 | No fallback — requires structured `decisionPoints` always | Partial | Clean but breaks legacy payloads |
| AIB-350 | Fallback via `buildDecisionPointsFallback` when array empty/absent | Yes | Safe, explicit naming |

**Best**: AIB-350/AIB-353 (tie) — Both preserve backwards compatibility. AIB-352's removal of fallback is cleaner but could break existing integrations.

### Decision Point 3: Input Validation Depth
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-353 | Zod schemas only (no cross-field validation) | Partial | Simple but allows invalid references |
| AIB-352 | Zod `superRefine` + route-level validation for verdict keys against participants | Yes | Most robust, prevents invalid data at boundary |
| AIB-350 | Basic Zod schema, no cross-referencing | No | Weakest validation |

**Best**: AIB-352 — Validates verdict ticket keys reference actual participants at both schema and route level, per Security-First principle.

### Decision Point 4: Scope of Changes
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-353 | Types + payload + record builder + tests | Yes | Focused, minimal |
| AIB-352 | Types + payload + record + route + component + generator + detail + fixtures + tests | Yes | Complete end-to-end |
| AIB-350 | Types + payload + record + tests | Yes | Minimal |

**Best**: AIB-352 — Touches all layers: API route validation, component rendering (empty approaches), detail sorting, markdown generation, and test fixtures. True full-stack implementation.

## Ranking & Recommendation

### 1. **AIB-352** - Best code (score: 93%)
**Why it wins**:
- Most comprehensive: 8 source + 8 test files covering all layers
- Route-level validation prevents invalid verdict keys (Security-First)
- Zod `superRefine` cross-references decision point tickets against participants
- Component defensive rendering for empty approaches
- `participantApproaches` naming matches DB schema
- DRY extraction of `normalizeDecisionPointApproaches` and `buildDecisionPointApproaches`
- Sparse historical row test ensures backwards-compatible reads

### 2. **AIB-353** - Good, focused (score: 84%)
**Why it ranks second**:
- Clean structured path with fallback preservation
- Good Zod schema design with proper type interfaces
- Missing route-level validation and component updates
- Fewer touchpoints — doesn't update generator or detail layer

### 3. **AIB-350** - Minimal (score: 72%)
**Why it ranks last**:
- `Record<string, string>` for approaches loses ordering guarantee
- `winner` field name doesn't match DB `verdictTicketId` semantics
- `optional()` without `.default([])` means consumers must null-check
- No cross-field Zod validation
- Only 2 test files, no integration tests for the new feature
