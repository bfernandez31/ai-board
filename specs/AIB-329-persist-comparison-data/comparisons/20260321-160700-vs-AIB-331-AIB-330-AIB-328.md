## Implementation Choices Analysis

### Decision Point 1: API Route Structure
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | POST on `/tickets/:id/comparisons` | Yes | Proper REST nesting under ticket |
| AIB-331 | POST on `/tickets/:id/comparisons` | Yes | Same approach as AIB-329 |
| AIB-330 | POST on `/tickets/:id/comparisons` | Yes | Same route + adds idempotency via compareRunKey, duplicate detection (200 vs 201) |
| AIB-328 | POST on `/projects/:projectId/comparisons` | Partial | Project-level route, no ticket scoping |

**Best**: AIB-330 — Ticket-scoped route with idempotency handling for workflow retries.

### Decision Point 2: Validation & Type Safety
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | Inline Zod schemas in route, `as` casts for enums | Good | Functional but duplicates type concerns |
| AIB-331 | Inline Zod with `z.nativeEnum`, helper functions for normalization | Better | Cleaner enum handling, reusable normalizers |
| AIB-330 | Dedicated `comparison-payload.ts` module with serialization/deserialization types | Best | Proper separation, reusable schemas, explicit serialization boundary |
| AIB-328 | `z.looseObject({})` for nested report fields | Weak | Loose validation allows malformed data through |

**Best**: AIB-330 — Dedicated payload module with strict schemas and explicit serialization types.

### Decision Point 3: Data Persistence Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | Route calls `persistGeneratedComparisonArtifacts` directly | OK | Tight coupling between route and generator |
| AIB-331 | Similar direct call with fixture builder helper | OK | Better test ergonomics |
| AIB-330 | Generator produces JSON artifact → workflow POSTs to API → API calls `persistComparisonRecord` | Best | Decoupled: generator doesn't touch DB, API handles persistence |
| AIB-328 | Route calls `persistComparisonRecord` with `as unknown as` cast | Weak | Type-unsafe cast bypasses validation |

**Best**: AIB-330 — Clean separation: generator produces artifact, API persists. No unsafe casts.

### Decision Point 4: Idempotency / Duplicate Handling
| Ticket | Approach | Notes |
|--------|----------|-------|
| AIB-329 | None | Re-POST creates duplicate records |
| AIB-331 | None | Same issue |
| AIB-330 | `compareRunKey` field + Prisma migration + transaction-level dedup | Returns 200 for duplicates vs 201 for new |
| AIB-328 | None | Same issue |

**Best**: AIB-330 — Only implementation with idempotency support, critical for workflow retries.

### Decision Point 5: Workflow Integration
| Ticket | Approach | Notes |
|--------|----------|-------|
| AIB-329 | curl POST with response parsing | Solid error handling |
| AIB-331 | curl POST with `-f` flag | Simpler but less diagnostic |
| AIB-330 | curl POST + jq validation + artifact cleanup + status-aware responses | Most robust: validates JSON, cleans up, differentiates created/duplicate/error |
| AIB-328 | curl POST + `ls -t *.json` to find file | Fragile: depends on file ordering |

**Best**: AIB-330 — Most defensive workflow step with proper error differentiation.

## Ranking & Recommendation

### 1. **AIB-330** - Best code (score: 88%)
**Why it wins**:
- Strongest architecture: dedicated payload module with serialization/deserialization
- Only implementation with idempotency (`compareRunKey` + Prisma migration)
- Best test coverage: 6 test files covering unit and integration layers
- Strict Zod validation with proper type boundaries
- Clean separation between artifact generation and persistence
- Most robust workflow integration step

### 2. **AIB-331** - Runner-up (score: 80%)
**Why it lost**:
- No idempotency handling
- No dedicated payload module — schemas inline in route
- Good but fewer tests (2 files)

### 3. **AIB-329** - Source (score: 78%)
**Why it lost**:
- No idempotency handling
- Enum `as` casts instead of native enum validation
- Single test file
- Direct coupling to generator function

### 4. **AIB-328** - Weakest (score: 65%)
**Why it lost**:
- Project-level route instead of ticket-scoped
- `z.looseObject({})` allows malformed nested data
- Unsafe `as unknown as` type cast
- Fragile workflow step (`ls -t *.json | head -1`)
- No idempotency support
