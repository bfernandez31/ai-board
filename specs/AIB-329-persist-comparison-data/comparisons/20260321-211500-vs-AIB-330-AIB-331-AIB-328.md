## Implementation Choices Analysis

### Decision Point 1: API Route Structure
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | POST on `/tickets/[id]/comparisons` | Yes | Consistent with existing GET |
| AIB-330 | No code | N/A | N/A |
| AIB-331 | POST on `/tickets/[id]/comparisons` | Yes | Consistent with existing GET, extra source ticket validation |
| AIB-328 | POST on `/projects/[projectId]/comparisons` | Partial | Different level than existing ticket-scoped GET |

**Best**: AIB-331 & AIB-329 - Ticket-scoped route matches existing GET handler. AIB-328 puts POST at project level which breaks REST consistency.

### Decision Point 2: Zod Validation Strictness
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | Full strict Zod schemas for all nested objects | Yes | Complete input validation |
| AIB-331 | Strict Zod with `z.nativeEnum()` for Prisma enums | Yes | Stronger type-Prisma alignment |
| AIB-328 | `z.looseObject({})` for alignment/implementation/compliance | No | Weak validation, any data passes |

**Best**: AIB-331 - `nativeEnum(Stage)` ensures Zod validation stays in sync with Prisma schema automatically. AIB-328's `z.looseObject({})` violates Security-First principle by accepting arbitrary data.

### Decision Point 3: Type Safety & Casting
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | Explicit type transformation with local `TicketRef` type | Yes | Clear, safe |
| AIB-331 | Helper functions `normalizeImplementationMetrics()` | Yes | Reusable, testable |
| AIB-328 | `as unknown as` casting for report type | No | Unsafe, bypasses TypeScript |

**Best**: AIB-331 - Modular normalization functions are safer and more maintainable than `as unknown as` casts.

### Decision Point 4: Source Ticket Validation
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | Verifies ticket exists & belongs to project | Yes | Basic safety |
| AIB-331 | Verifies ticket + validates sourceTicket.id AND ticketKey match route | Yes | Prevents payload/route mismatch |
| AIB-328 | Validates projectId body/URL match only | Partial | No ticket verification |

**Best**: AIB-331 - Double validation (id + ticketKey) prevents subtle bugs where payload references wrong ticket.

### Decision Point 5: Test Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-329 | 1 integration test file (3 cases) | Partial | Covers happy path + auth + validation |
| AIB-331 | 3 files: integration (extended existing) + unit + shared fixtures | Yes | Reusable fixtures, extends existing test file |
| AIB-328 | 1 integration test file (8 cases) | Yes | Thorough but standalone, manual cleanup |

**Best**: AIB-331 - Extends existing test file (no duplication), adds unit tests, creates reusable `buildGeneratedComparisonArtifactPayload` fixture.

## Ranking & Recommendation

### 1. **AIB-331** - Best code (score: 90%)
**Why it wins**:
- Strongest type safety: `z.nativeEnum()` auto-syncs with Prisma enums
- Modular normalization helpers (testable, reusable)
- Source ticket double-validation (id + ticketKey)
- Best test architecture: extends existing tests, shared fixtures, unit test coverage
- Most cost-efficient ($4.92, 25m, 2 jobs)

### 2. **AIB-329** - Good implementation (score: 84%)
**Why it lost**:
- Less test coverage (1 file vs 3)
- No source ticket key validation
- Inline type transformation instead of reusable functions

### 3. **AIB-328** - Weakened validation (score: 73%)
**Why it lost**:
- `z.looseObject({})` accepts arbitrary data (security concern)
- `as unknown as` type casting bypasses TypeScript safety
- Project-level endpoint breaks REST consistency with existing ticket-scoped GET
- Most expensive ($19.78, 54m)

### 4. **AIB-330** - No code (score: 0%)
**Why it lost**:
- Zero code changes on branch
