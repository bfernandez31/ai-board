## Implementation Choices Analysis

### Decision Point 1: Metric Computation Location
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-354 | Client-side (ComparisonUnifiedMetrics computes inline) | Partial | Simpler, but logic duplicated in component |
| AIB-355 | Server-side (`normalizeDashboardDetail` in comparison-detail.ts) | Yes | Centralized, reusable, testable via integration tests |

**Best**: AIB-355 - Server-side normalization keeps components pure renderers, aligns with Next.js App Router philosophy.

### Decision Point 2: Component Strategy
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-354 | Replace components (new HeroCard, ScoreGauge, StatCards, ParticipantGrid) | Yes | Clean slate, more visual innovation, but high churn |
| AIB-355 | Evolve existing components (modify ComparisonRanking, OperationalMetrics in-place) | Yes | Lower risk, iterative, maintains API surface |

**Best**: AIB-355 - Iterative evolution reduces regression risk while still delivering the redesign.

### Decision Point 3: Color Token Usage
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-354 | Uses `bg-ctp-green/20`, `bg-ctp-yellow/20`, `bg-ctp-red/20` | Warning | Custom catppuccin tokens — may not be standard semantic tokens |
| AIB-355 | Uses `bg-primary/10`, `bg-destructive/10`, `bg-accent/60` | Yes | Standard Tailwind semantic tokens per constitution |

**Best**: AIB-355 - Constitution explicitly requires semantic tokens (`text-foreground`, `bg-card`, etc.), not custom palette tokens.

### Decision Point 4: Type System Richness
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-354 | New prop types (HeroCardProps, StatCardsProps, etc.) per component | Yes | Focused, but types tied to specific components |
| AIB-355 | Shared dashboard metric types (`ComparisonDashboardMetricRow`, `ComparisonDashboardMetricCell`) | Yes | Reusable across headline + detail metrics, server-computed |

**Best**: AIB-355 - Shared types with `category: 'headline' | 'detail'` enable flexible rendering without duplicating type definitions.

### Decision Point 5: Testing Strategy
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-354 | 10 unit test files (component-focused), no integration tests | Partial | High component coverage, no backend validation |
| AIB-355 | 6 test files (unit + integration), updated comparison-fixtures.ts | Yes | Integration tests validate server normalization; fixture upgrades benefit all tests |

**Best**: AIB-355 - Integration tests are more valuable per constitution's test decision tree (API/DB → integration).

## Ranking & Recommendation

### 1. **AIB-355** - Best code (score: 82%)
**Why it wins**:
- Server-side metric normalization (`normalizeDashboardDetail`) keeps components as pure renderers
- Proper semantic Tailwind tokens (primary, destructive, accent) per constitution
- Integration tests validating backend normalization logic
- Shared `ComparisonDashboardMetricRow` type system enables flexible metric rendering
- `placeholderData` in TanStack Query improves UX during navigation
- Lower cost ($9.94 vs $13.36)

### 2. **AIB-354** - Source (score: 78%)
**Why it lost**:
- Client-side metric computation in components (less maintainable)
- Uses `bg-ctp-*` tokens instead of semantic Tailwind tokens
- No integration tests (only unit tests)
- However: Superior visual innovation (ScoreGauge SVG, HeroCard, StatCards with micro-bars, compliance heatmap)

**Notable strengths of AIB-354 worth preserving**:
- ScoreGauge component (animated SVG ring) is excellent
- HeroCard with stat pills provides strong visual hierarchy
- Compliance heatmap with tooltip is more compact than grid
- ParticipantGrid card layout is more scannable
