## Implementation Choices Analysis

### Decision Point 1: Component Architecture

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-277 | 3 components (card, faq, section) | Yes | Clean separation, reusable card |
| AIB-278 | 3 components + data layer (`lib/landing/pricing.ts`) | Best | Types + constants centralized, most extensible |
| AIB-279 | Single 182-line file with everything inline | Partial | Simple but monolithic, violates component-driven |
| AIB-280 | Single 123-line file, minimal | Partial | Compact but not maintainable at scale |

**Best**: AIB-278 - Proper feature folder with dedicated data layer, typed interfaces (`LandingPricingPlan`, `LandingPricingEmphasis`), and exported constants for cross-component use.

### Decision Point 2: Data Source for Plan Config

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-277 | Imports `PLANS` from `lib/billing/plans` | Yes | Single source of truth |
| AIB-278 | Wraps `PLANS` via `lib/landing/pricing.ts` adapter | Best | Decoupled, landing-specific shaping |
| AIB-279 | **Hardcoded** prices ($29, $99) and features | No | Breaks when billing config changes |
| AIB-280 | Imports `PLANS` from `lib/billing/plans` | Yes | Single source of truth |

**Best**: AIB-278 - Adapter pattern wraps billing plans into landing-specific shape without coupling UI directly to billing internals.

**Worst**: AIB-279 - Hardcoded plan data will drift from actual billing config. Pro shows $29 instead of the canonical $15.

### Decision Point 3: Header/Footer Integration

| Ticket | Approach | Notes |
|--------|----------|-------|
| AIB-277 | No integration | Pricing section only, no nav links |
| AIB-278 | Full integration | Pricing link in header nav + footer, with anchor href constant |
| AIB-279 | Footer only | Refactored footer to array-based links, added pricing |
| AIB-280 | No integration | Pricing section only |

**Best**: AIB-278 - Complete navigation integration with shared anchor constant.

### Decision Point 4: FAQ Implementation

| Ticket | Approach | Notes |
|--------|----------|-------|
| AIB-277 | Interactive collapsible (client component) | `'use client'` + useState, Radix Collapsible |
| AIB-278 | Static cards (server component) | No JS needed, grid layout |
| AIB-279 | Static cards (server component) | Similar to AIB-278 |
| AIB-280 | Static divs (server component) | Simplest |

**Trade-off**: AIB-277's interactive FAQ is more engaging but requires client-side JS. AIB-278/279's static approach is simpler and works as server component. Both valid choices.

### Decision Point 5: Type Safety & Return Types

| Ticket | Approach |
|--------|----------|
| AIB-277 | Props interface, no explicit return types |
| AIB-278 | Full types, explicit `JSX.Element` returns, `SubscriptionPlan` import |
| AIB-279 | Inline type aliases, `React.JSX.Element` returns |
| AIB-280 | No explicit return types |

**Best**: AIB-278 - Strongest TypeScript discipline.

### Decision Point 6: Accessibility

| Ticket | `aria-hidden` on decorative icons | `data-testid` | Semantic HTML |
|--------|-------------------------------|---------------|---------------|
| AIB-277 | No | No | Yes |
| AIB-278 | Yes | Yes | Yes |
| AIB-279 | No | Yes | Yes |
| AIB-280 | No | Yes | Yes |

**Best**: AIB-278 - `aria-hidden="true"` on Check icons, proper test IDs.

## Ranking & Recommendation

### 1. **AIB-278** - Best code (score: 93%)
**Why it wins**:
- Dedicated data layer (`lib/landing/pricing.ts`) with typed interfaces and exported constants
- Full header + footer navigation integration with shared anchor href
- Best test coverage: 5 test files covering pricing section, landing page order, header nav, footer links, and data utilities
- Strongest TypeScript: explicit return types, `SubscriptionPlan` enum import, adapter pattern
- Accessibility: `aria-hidden` on decorative icons, `data-testid` for all testable elements
- Updated consolidated specs

### 2. **AIB-277** (source) - Good (score: 83%)
**Why it placed second**:
- Clean 3-component architecture with proper separation
- Interactive FAQ (collapsible) adds engagement value
- Comprehensive tests (155 lines) but only 1 file
- Missing: no header/footer integration, no explicit return types, no data layer abstraction

### 3. **AIB-279** - Adequate (score: 66%)
**Why it lost**:
- **Critical flaw**: Hardcoded plan prices ($29/$99) diverge from canonical PLANS config ($15/$30)
- Monolithic 182-line single file violates component-driven principle
- Footer refactor is clean but insufficient to offset data integrity issue
- Reasonable test ratio but shallow coverage

### 4. **AIB-280** - Incomplete (score: 48%)
**Why it lost**:
- Zero test files - violates constitution Principle III (Test-Driven)
- Minimal scope: only 2 source files, no header/footer integration
- Uses PLANS correctly but lacks any abstraction or type enrichment
- No consolidated spec updates
