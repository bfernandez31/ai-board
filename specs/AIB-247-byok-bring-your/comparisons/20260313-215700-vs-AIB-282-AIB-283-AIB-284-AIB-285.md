## Implementation Choices Analysis

### Decision Point 1: State Management (UI Component)

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-247 | useState + useEffect + useCallback | Partial | Manual fetching, no cache invalidation |
| AIB-282 | useState + useEffect + useCallback | Partial | Same manual approach |
| AIB-283 | useState + useEffect + useCallback | Partial | Manual fetch, no TanStack Query |
| AIB-284 | useState + useRouter.refresh() | Partial | Server-side refresh, no client cache |
| AIB-285 | useQuery + useMutation (TanStack Query) | **Yes** | Proper cache invalidation, loading/error states |

**Best**: AIB-285 - Uses TanStack Query per constitution (codebase standard for server state).

### Decision Point 2: Encryption Architecture

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-247 | Single `encrypt`/`decrypt` in `lib/crypto/encryption.ts`, combined encrypted string | **Yes** | Clean, single utility |
| AIB-282 | `encryptApiKey` returns separate iv/authTag/encryptedKey fields | Partial | Separate DB columns for crypto components |
| AIB-283 | `lib/security/project-ai-credentials.ts` with service layer | **Yes** | Over-engineered service pattern |
| AIB-284 | Inline on Project model (columns on project table) | Partial | No separate table, tighter coupling |
| AIB-285 | `lib/encryption/api-keys.ts` with separate `lib/types/api-keys.ts` | **Yes** | Clean separation of concerns |

**Best**: AIB-247/AIB-285 tied - Both use AES-256-GCM with clean abstractions.

### Decision Point 3: API Route Design

| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-247 | `/api-keys` (GET/POST) + `/api-keys/[provider]` (DELETE) + `/api-keys/[provider]/validate` | **Yes** | RESTful, provider in URL for actions |
| AIB-282 | `/api-keys` (GET/PUT/DELETE) + `/api-keys/validate` | Partial | DELETE requires body with provider |
| AIB-283 | `/ai-credentials/[provider]` (PUT/DELETE) + `/ai-credentials` (GET) + `/jobs/[jobId]/provider-credentials` | **Yes** | Different naming, includes workflow credential endpoint |
| AIB-284 | `/api-keys` (GET/PATCH/DELETE) + `/api-keys/test` + `/agent-credentials` | **Yes** | Includes agent-credentials for workflows |
| AIB-285 | `/api-keys` (GET/POST) + `/api-keys/[provider]` (DELETE) + `/api-keys/validate` | **Yes** | RESTful, includes `skipValidation` option |

**Best**: AIB-285 - RESTful design with `skipValidation` flag for unreachable providers (FR-014 compliance).

### Decision Point 4: Workflow Integration

| Ticket | Approach | Constitution | Trade-offs |

## Ranking & Recommendation

### 1. **AIB-285** - Best code (score: 90%)
**Why it wins**:
- Highest test coverage: 7 test files (unit, integration, component) covering CRUD, validation, workflow dispatch, and UI
- TanStack Query for proper server state management (only ticket to do this)
- Explicit TypeScript types in `lib/types/api-keys.ts`
- AlertDialog for delete confirmation
- `skipValidation` flag for unreachable providers
- Clean separation: validation lib, encryption lib, types lib, DB layer

**Weakness**: No workflow file changes (would need AIB-283/284's approach for full integration)

### 2. **AIB-247** (source) - Strong runner-up (score: 87%)
**Why it's second**:
- Clean encryption abstraction with combined string format
- Good test coverage (4 files including component test)
- AlertDialog for destructive actions
- Owner-only access properly enforced with `isOwner` prop

**Weakness**: useState instead of TanStack Query; workflow inputs expose keys in dispatch payload

### 3. **AIB-283** - Good architecture (score: 82%)
**Why it's third**:
- Best workflow integration (all 6 workflow files updated + run-agent.sh)
- Service layer pattern with `ai-credential-service.ts`
- Provider validation service
- 5 test files including transition-byok test

**Weakness**: Over-engineered service layer; non-standard naming (`ai-credentials` vs `api-keys`); no TanStack Query; no AlertDialog confirmed

### 4. **AIB-284** - Efficient but incomplete (score: 75%)
**Why it's fourth**:
- Cleanest workflow integration via run-agent.sh `fetch_project_credentials`
- Inline columns on Project model (simpler but less scalable)
- Agent-credentials endpoint for workflow use

**Weakness**: Keys on Project table (not separate model); no AlertDialog for delete; only 3 test files; `useState` only; modified too many spec files

### 5. **AIB-282** - Minimal implementation (score: 72%)
**Why it's last**:
- Fewest source files (11) and tests (2)
- No workflow integration at all
- DELETE via body (non-RESTful)
- Uses PUT instead of POST for create
- No delete confirmation dialog

**Weakness**: Incomplete feature (no workflow injection); minimal test coverage
