# Data Model: Health Dashboard — Scan Detail Drawer

**Branch**: `AIB-371-health-dashboard-drawer` | **Date**: 2026-03-29

## Overview

This feature does **not** add new Prisma models or database migrations. It operates on the existing `HealthScan` and `HealthScore` models, consuming the `report` field (JSON string) with a newly defined client-side schema.

## Existing Entities (Read-Only)

### HealthScan

Already exists in `prisma/schema.prisma`. The drawer reads these fields:

| Field | Type | Drawer Usage |
|-------|------|-------------|
| `id` | Int | Identifies the scan for history |
| `scanType` | HealthScanType | Determines module and grouping strategy |
| `status` | HealthScanStatus | Determines drawer state (scanning, completed, failed) |
| `score` | Int? | Displayed in header and history |
| `report` | String? | **Primary content** — parsed as structured JSON for issue display |
| `issuesFound` | Int? | Summary count in header and history |
| `issuesFixed` | Int? | Shown in Tests module for auto-fixed count |
| `baseCommit` | String? | Commit range display |
| `headCommit` | String? | Commit range display |
| `errorMessage` | String? | Shown for failed scans |
| `completedAt` | DateTime? | Last scan date in header |
| `createdAt` | DateTime | History ordering |

### HealthScore

Already exists. Provides aggregate scores for the hero and module cards. Not directly used by the drawer beyond what the health endpoint already returns.

## New Client-Side Types (TypeScript Only)

These types define the structure of the `report` JSON field. They live in `lib/health/types.ts` alongside existing types.

### Common Types

```typescript
/** Individual issue found by a scan */
interface ReportIssue {
  id: string;
  severity: 'high' | 'medium' | 'low';
  description: string;
  file?: string;
  line?: number;
  category?: string;
}

/** Ticket generated by a scan */
interface GeneratedTicket {
  ticketKey: string;
  stage: string;
}
```

### Module-Specific Report Variants

```typescript
/** Security: issues grouped by severity */
interface SecurityReport {
  type: 'SECURITY';
  issues: ReportIssue[];
  generatedTickets: GeneratedTicket[];
}

/** Compliance: issues grouped by category (constitution principle) */
interface ComplianceReport {
  type: 'COMPLIANCE';
  issues: ReportIssue[];
  generatedTickets: GeneratedTicket[];
}

/** Tests: split into auto-fixed and non-fixable */
interface TestsReport {
  type: 'TESTS';
  autoFixed: ReportIssue[];
  nonFixable: ReportIssue[];
  generatedTickets: GeneratedTicket[];
}

/** Spec Sync: per-spec sync status */
interface SpecSyncReport {
  type: 'SPEC_SYNC';
  specs: SpecSyncEntry[];
  generatedTickets: GeneratedTicket[];
}

interface SpecSyncEntry {
  specPath: string;
  status: 'synced' | 'drifted';
  drift?: string;
}

/** Quality Gate (passive): dimension breakdown */
interface QualityGateReport {
  type: 'QUALITY_GATE';
  dimensions: QualityDimension[];
  recentTickets: QualityTicket[];
}

interface QualityDimension {
  name: string;
  score: number | null;
}

interface QualityTicket {
  ticketKey: string;
  score: number | null;
}

/** Last Clean (passive): cleanup summary */
interface LastCleanReport {
  type: 'LAST_CLEAN';
  filesCleaned: number;
  remainingIssues: number;
  summary: string;
}

/** Discriminated union of all report types */
type ScanReport = SecurityReport | ComplianceReport | TestsReport | SpecSyncReport | QualityGateReport | LastCleanReport;
```

### Zod Validation Schemas

Corresponding Zod schemas will be created for runtime validation of parsed report JSON. On parse failure, the drawer renders a fallback message.

## State Transitions

No new state transitions. The drawer is a read-only view that displays data based on the existing `HealthScanStatus`:

| Status | Drawer Behavior |
|--------|----------------|
| `null` (never scanned) | "No scan yet" message + scan trigger button |
| `PENDING` / `RUNNING` | Scanning indicator with animation |
| `COMPLETED` | Full report display with issues, tickets, history |
| `FAILED` | Error message with `errorMessage` field content |

## Relationships

```
HealthScan.report (JSON String)
    └── Parsed as ScanReport (discriminated on scanType)
         ├── SecurityReport.issues[] → grouped by severity
         ├── ComplianceReport.issues[] → grouped by category (principle)
         ├── TestsReport.autoFixed[] / nonFixable[]
         ├── SpecSyncReport.specs[] → grouped by sync status
         ├── QualityGateReport.dimensions[] + recentTickets[]
         └── LastCleanReport → summary view

GeneratedTicket.ticketKey → links to /projects/{projectId}/board (ticket navigation)
```

## Validation Rules

1. `report` field: Validated with Zod at render time; graceful fallback on failure
2. `GeneratedTicket.ticketKey`: Must match `{PROJECT_KEY}-{NUMBER}` format for link generation
3. `ReportIssue.severity`: Restricted to 'high' | 'medium' | 'low' via Zod enum
4. Score values: 0-100 integer range (enforced by existing API)
