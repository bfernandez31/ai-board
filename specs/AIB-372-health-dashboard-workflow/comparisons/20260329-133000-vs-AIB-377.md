## Implementation Choices Analysis

### [Spec] US3: Ticket grouping by scan-type rules
| Ticket | Approach |
|--------|----------|
| AIB-372 | All grouping logic inline in bash workflow (curl + jq) |
| AIB-377 | Typed TypeScript module `lib/health/ticket-creation.ts` with `groupIssuesIntoTickets()` + bash workflow for HTTP calls |

**Best**: AIB-377 - Extracts business logic into testable TypeScript. Grouping rules have comprehensive unit tests (181 lines). The bash workflow still handles HTTP, but the logic is type-safe and reusable server-side.

### [Spec] US5: Telemetry recording
| Ticket | Approach |
|--------|----------|
| AIB-372 | Full OTEL env vars (`OTEL_LOGS_EXPORTER`, `OTEL_EXPORTER_OTLP_PROTOCOL`, etc.) + duration/tokens/cost capture |
| AIB-377 | Basic duration/tokens/cost capture, no OTEL config |

**Best**: AIB-372 - OTEL integration enables centralized observability. AIB-377 only captures basic metrics.

### Decision Point 1: Command Mapping Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-372 | `case` statement in bash only | Partial (no TS types) | Simple but not reusable |
| AIB-377 | `Record<HealthScanType, string>` in `lib/health/scan-commands.ts` + bash `case` | Yes (TypeScript-First) | Type-safe, testable, reusable |

**Best**: AIB-377 - Typed mapping with unit tests follows Principle I (TypeScript-First). The `getScanCommand()` function can be used by API routes or future server-side dispatchers.

### Decision Point 2: Ticket Creation Auth in API Route
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-372 | `validateWorkflowAuth()` → skip plan limits → validate project exists | Yes | Proper dual-auth |
| AIB-377 | Same pattern, nearly identical code | Yes | Same approach |

**Best**: Tie - Both implementations are functionally equivalent and properly handle the dual-auth pattern.

### Decision Point 3: Workflow Error Handling
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-372 | Separate steps for scan error vs parse error, dedicated error messages | Yes (Security-First) | More granular failure modes |
| AIB-377 | Single failure path, simpler error handling | Partial | Less diagnostic info |

**Best**: AIB-372 - Distinguishing parse errors from scan errors improves debugging.

### Decision Point 4: Component Refactoring
| Ticket | Approach |
|--------|----------|
| AIB-372 | `!!module.passive` instead of `=== true` |
| AIB-377 | Same + moves `BUTTON_LABELS` to module scope |

**Best**: AIB-377 - Module-scoped constant avoids recreation on every render.

### Decision Point 5: Plugin Command Definitions
| Ticket | Approach |
|--------|----------|
| AIB-372 | No scan command definitions included |
| AIB-377 | 4 `.claude-plugin/commands/` files with JSON output schemas |

**Best**: AIB-377 - Includes the command interface contracts that the workflow depends on. Without these, the scan commands have no specification.

| Ticket | I. TypeScript | II. Components | III. Tests | IV. Security | V. Database | Overall |
