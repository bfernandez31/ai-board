## Implementation Choices Analysis

### [Spec] Workflow token auth for ticket creation (FR-009)
| Ticket | Approach |
|--------|----------|
| AIB-372 | Modifies `tickets/route.ts` to accept workflow token auth, skipping subscription limits, validating project exists |
| AIB-377 | Sends `Authorization: Bearer` header but does NOT modify the endpoint — ticket creation would fail at auth |

**Best**: AIB-372 — Without API auth modification, AIB-377's ticket creation is non-functional.

### [Spec] Agent execution model (FR-014)
| Ticket | Approach |
|--------|----------|
| AIB-372 | Uses existing `run-agent.sh` with `agent` input parameter supporting CLAUDE and CODEX |
| AIB-377 | Installs Claude Code via npm directly, no agent input, no Codex support |

**Best**: AIB-372 — Follows existing speckit pattern, supports both agents.

### [Spec] Error handling for multiple failure modes (FR-008)
| Ticket | Approach |
|--------|----------|
| AIB-372 | 3 distinct failure steps: scan error, parse error, unexpected failure (with `if: failure()`) |
| AIB-377 | Single failure step, no parse error distinction, no unexpected failure handler |

**Best**: AIB-372 — More comprehensive error handling per spec.

### Decision Point 1: Code Separation (Ticket Logic)
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-372 | Ticket creation logic inline in YAML workflow | N/A (workflow) | Pragmatic, all logic in one place |
| AIB-377 | TypeScript modules (`scan-commands.ts`, `ticket-creation.ts`) + duplicate in YAML | Yes (I. TypeScript) | Better separation but modules unused by workflow |

**Best**: AIB-377 has better code architecture with TS modules, but the modules aren't actually invoked by the workflow (the workflow duplicates the logic in bash/jq). AIB-372's inline approach is more honest — the logic lives where it runs.

### Decision Point 2: Telemetry Setup

| AIB-377 | Basic SECONDS timing, optional token/cost extraction | N/A | Minimal telemetry |

**Best**: AIB-372 — OTEL integration follows existing workflow patterns.

### Decision Point 3: Repository Checkout
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
