## Implementation Choices Analysis

### [Spec] Quality scores in ticket selection UI
| Ticket | Approach |
|--------|----------|
| AIB-358 | Shows ticketKey, title, and branch — no quality score |
| AIB-362 | Shows ticketKey, title, and quality score badge (available/pending/N/A) via dedicated candidates API |

**Best**: AIB-362 — Fully implements FR-007 with enriched quality data from a candidates endpoint.

### [Spec] Pending launch state tracking (FR-010)
| Ticket | Approach |
|--------|----------|
| AIB-358 | No pending state tracking; launch fires and dialog closes |
| AIB-362 | Tracks pending launches in state, polls job status via `refetchInterval`, shows in-progress banner, invalidates list on terminal status |

**Best**: AIB-362 — Complete launch lifecycle feedback.

### Decision Point 1: Hook Architecture
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-358 | New standalone `use-project-comparisons.ts` hook (153 lines) | Yes | Clean but duplicates fetch helpers and key patterns already in `use-comparisons.ts` |
| AIB-362 | Extends existing `use-comparisons.ts` with new query keys and hooks (+109 lines) | Yes | DRY, single source of truth for comparison queries, shared fetch helper |

**Best**: AIB-362 — Extends existing patterns, avoids duplication.

### Decision Point 2: Lib-Layer Separation
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-358 | Business logic inline in API routes | Yes | Simpler but harder to test in isolation |
| AIB-362 | Dedicated lib files: `project-comparison-candidates.ts`, `project-comparison-launch.ts`, `project-comparison-summary.ts` + refactored shared `getComparisonDetailByWhere` | Yes | Testable, reusable, follows feature-folder principle |

**Best**: AIB-362 — Proper separation of concerns with reusable lib functions.

### Decision Point 3: Layout Pattern
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-358 | Vertical list with inline expand/collapse below each item | Yes | Simple, familiar list pattern |
| AIB-362 | 2-column layout: sidebar history list (340px) + detail panel | Yes | More efficient screen usage, avoids list scroll disruption when viewing detail |

**Best**: AIB-362 — Better UX for desktop; keeps list visible while viewing detail.

### Decision Point 4: Launcher Component
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-358 | Dialog (modal overlay) with checkboxes | Yes | Simple, focused interaction |
| AIB-362 | Sheet (slide-out panel) with card selection + pending launch display | Yes | Richer interaction, shows pending state, doesn't obscure page |

**Best**: AIB-362 — Sheet pattern provides more space and context.

### Decision Point 5: VERIFY Tickets Data Source
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-358 | Reuses existing `/tickets/verify` route (modified for dual auth) | Yes | Pragmatic reuse, but couples with auto-ship endpoint |
| AIB-362 | New `/comparisons/candidates` endpoint with quality score enrichment | Yes | Purpose-built, returns enriched data specific to comparison needs |

**Best**: AIB-362 — Purpose-built endpoint avoids coupling and provides quality scores.
