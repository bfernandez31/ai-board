## Implementation Choices Analysis

### [Spec] Page-based pagination
| Ticket | Approach |
|--------|----------|
| AIB-362 | Page/pageSize params with Previous/Next controls and page indicator |
| AIB-358 | Offset-based "Load More" pattern with infinite accumulation via useRef |

**Best**: AIB-362 - Matches spec auto-resolved decision for "page-by-page loading" and avoids memory accumulation.

### [Spec] Quality score display on candidates
| Ticket | Approach |
|--------|----------|
| AIB-362 | Dedicated `/comparisons/candidates` API returning quality score state (available/pending/N/A) |
| AIB-358 | Reuses `/tickets/verify` endpoint; no quality score shown in launcher |

**Best**: AIB-362 - Fulfills FR-010 requiring quality score state per candidate.

### [Spec] Pending launch tracking
| Ticket | Approach |
|--------|----------|
| AIB-362 | Client-side pending launches with 2s job polling, auto-refetch on completion |
| AIB-358 | No visible pending state tracking after launch |

**Best**: AIB-362 - Fulfills FR-013 requiring visible pending state.

### Decision Point 1: Code Organization
| Ticket | Approach | Constitution | Trade-offs |

**Best**: AIB-362 - Constitution II requires feature-based folder structure. Creating a parallel folder with near-identical name violates discoverability.
