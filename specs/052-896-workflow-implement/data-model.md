# Data Model: Enhanced Implementation Workflow

**Feature**: Database Setup and Selective Testing for `/speckit.implement` Command
**Date**: 2025-10-25
**Branch**: 052-896-workflow-implement

## Overview

This feature does **NOT introduce new database entities or schema changes**. It enhances the GitHub Actions workflow infrastructure to support database-dependent testing during implementation.

The existing data model remains unchanged. This document describes how existing entities are utilized during the enhanced workflow.

### Job Model
**Purpose**: Tracks GitHub Actions workflow execution status
**Location**: `prisma/schema.prisma`
**Usage in Feature**: Jobs created for implement command, status updated by workflow

**Existing Schema**:
```prisma
model Job {
  id          Int       @id @default(autoincrement())
  ticketId    Int
  projectId   Int
  command     String
  status      JobStatus @default(PENDING)
  branch      String?   @db.VarChar(200)
  completedAt DateTime?
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  ticket      Ticket    @relation(fields: [ticketId], references: [id], onDelete: Cascade)
  project     Project   @relation(fields: [projectId], references: [id], onDelete: Cascade)

  @@index([ticketId])
  @@index([projectId])
  @@index([status])
}

enum JobStatus {
  PENDING
  RUNNING
  COMPLETED
  FAILED
  CANCELLED
}
```

**Workflow Interaction**:
- Job created with `command: "implement"` when user triggers workflow
- Status transitions: PENDING → RUNNING → COMPLETED/FAILED
- Updated via `PATCH /api/jobs/:id/status` endpoint

## Database Seeding (Enhanced Workflow Step)

### Test Fixtures Creation

**Script**: `tests/global-setup.ts` (called via `npx tsx tests/global-setup.ts`)
**Purpose**: Ensure test database has required fixtures before E2E tests

**Seeding Process**:
1. **Clean Database**: Remove leftover test data from previous runs
   - Delete all tickets from projects 1-2
   - Delete `[e2e]` prefixed tickets from other projects
   - Preserve projects 1, 2, 3 (stable fixtures + dev project)

2. **Create Test User**:
   ```typescript
   await prisma.user.upsert({
     where: { email: 'test@e2e.local' },
     update: {},
     create: {
       email: 'test@e2e.local',
       name: 'E2E Test User',
       emailVerified: new Date(),
     },
   });
   ```

3. **Create Test Projects**:
   ```typescript
   await prisma.project.upsert({
     where: { id: 1 },
     update: { userId: testUser.id },
     create: {
       id: 1,
       name: '[e2e] Test Project',
       githubOwner: 'test',
       githubRepo: 'test',
       userId: testUser.id,
     },
   });

   await prisma.project.upsert({
     where: { id: 2 },
     update: { userId: testUser.id },
     create: {
       id: 2,
       name: '[e2e] Test Project 2',
       githubOwner: 'test',
       githubRepo: 'test2',
       userId: testUser.id,
     },
   });
   ```

4. **Store Test User ID**:
   - Write to `.test-user-id` file for Playwright config
   - Used in `x-test-user-id` header for mock authentication

**No Changes to Seeding Logic**: Enhanced workflow calls existing seeding script, no modifications required.

## Migration Execution (Enhanced Workflow Step)

### Prisma Migration Process

**Command**: `npx prisma migrate deploy`
**Purpose**: Apply all pending migrations to test database

**Steps**:
1. **Generate Prisma Client**: `npx prisma generate`
   - Regenerates type-safe client from schema.prisma
   - Required before running migrations

2. **Apply Migrations**: `npx prisma migrate deploy`
   - Reads migration files from `prisma/migrations/`
   - Applies only unapplied migrations (idempotent)
   - Updates `_prisma_migrations` table to track applied migrations

3. **Validate Schema**: Prisma validates that database schema matches schema.prisma
   - Ensures no schema drift
   - Fails if manual schema changes detected

**DATABASE_URL Configuration** (workflow environment variable):
```bash
postgresql://postgres:postgres@localhost:5432/ai_board_test
```

**No Migration Files Added**: This feature doesn't add new migrations, only ensures existing migrations are applied before tests.

## State Transitions (No Changes)

### Job Status State Machine

**Existing Transitions** (enforced by `lib/job-state-machine.ts`):
```
PENDING → RUNNING → COMPLETED
                 ↘ FAILED
                 ↘ CANCELLED
```

**Workflow Update Pattern**:
```bash
# Workflow sets status to RUNNING when implement command starts
curl -X PATCH "${APP_URL}/api/jobs/${job_id}/status" \
  -H "Authorization: Bearer ${WORKFLOW_API_TOKEN}" \
  -d '{"status": "RUNNING"}'

# Workflow sets status to COMPLETED on success
curl -X PATCH "${APP_URL}/api/jobs/${job_id}/status" \
  -d '{"status": "COMPLETED"}'

# Workflow sets status to FAILED on error
curl -X PATCH "${APP_URL}/api/jobs/${job_id}/status" \
  -d '{"status": "FAILED"}'
```

**No Changes to State Machine**: Enhanced workflow uses existing status update API.

## Database Connection Management

### Connection String Format

**Test Database**:
```
postgresql://postgres:postgres@localhost:5432/ai_board_test
```

**Components**:
- **User**: `postgres` (default PostgreSQL user)
- **Password**: `postgres` (configured in service container)
- **Host**: `localhost` (service container mapped to runner)
- **Port**: `5432` (default PostgreSQL port)
- **Database**: `ai_board_test` (ephemeral test database)

### Connection Lifecycle

1. **Service Container Starts**: PostgreSQL service boots in background
2. **Health Check**: GitHub Actions waits for `pg_isready` to succeed
3. **Prisma Connects**: `prisma migrate deploy` connects to apply migrations
4. **Seeding Connects**: `tests/global-setup.ts` connects to create fixtures
5. **Claude Executes**: Implementation code can access database via DATABASE_URL
6. **Tests Execute**: Playwright tests connect to validate implementation
7. **Workflow Ends**: Service container destroyed, database wiped

**No Persistent State**: Database is ephemeral, recreated for each workflow run.
