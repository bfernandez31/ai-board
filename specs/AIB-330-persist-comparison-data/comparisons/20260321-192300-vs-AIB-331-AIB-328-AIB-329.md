## Implementation Choices Analysis

### Decision Point 1: Payload Serialization & Validation
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-330 | Dedicated `comparison-payload.ts` module with strict Zod schemas, serialization/deserialization functions | Yes - clean separation | Best: reusable, testable, strict |
| AIB-331 | Large inline Zod schema in route file with `normalizeImplementationMetrics` helper | Partial | Functional but monolithic |
| AIB-328 | `z.looseObject({})` for nested structures, `as unknown as` cast | No - weak typing | Loose validation defeats type safety |
| AIB-329 | Inline schemas in route, `as TicketRef` type assertions | Partial | Type casts bypass compiler checks |

**Best**: AIB-330 - Dedicated module with strict validation, proper serialization layer, no type assertions.

### Decision Point 2: Idempotency Handling
| Ticket | Approach | Constitution | Trade-offs |
|--------|----------|--------------|------------|
| AIB-330 | `compareRunKey` field + Prisma migration, duplicate detection returns 200 | Yes | Robust: workflow retries safe |
| AIB-331 | None - creates new record every POST | No | Duplicate records on retry |
| AIB-328 | None - creates new record every POST | No | Duplicate records on retry |
| AIB-329 | None - creates new record every POST | No | Duplicate records on retry |

**Best**: AIB-330 - Only implementation with idempotency support via `compareRunKey`.

## Ranking & Recommendation

### 1. **AIB-330** - Best code (score: 92%)
**Why it wins**:
- Dedicated `comparison-payload.ts` with strict Zod schemas and serialization layer
- Only ticket with `compareRunKey` idempotency (Prisma migration included)
- 9 test files covering unit (payload, generator, record) + integration (API, persistence, history)
- Most thorough route validation (path scope, key ordering, participant uniqueness)
- Clean helper extraction (`jsonError`, `parseRouteParams`, `orderParticipantsByPayload`)

### 2. **AIB-331** - (score: 78%)
**Why it's second**:
- Good test coverage (4 test files) with proper workflow auth testing
- Reuses existing `persistGeneratedComparisonArtifacts` (simple but couples API to internals)
- No idempotency handling
- Inline schema is large but functional

### 3. **AIB-329** - (score: 73%)
**Why it lost**:
- `as TicketRef` type assertions bypass TypeScript safety
- Only 1 test file with 4 basic test cases
- No idempotency handling
- Delegates to internal function designed for in-process use

### 4. **AIB-328** - (score: 68%)
**Why it lost**:
- `z.looseObject({})` provides weak validation for critical nested structures
- `as unknown as` type assertion is a code smell
- Project-level route (`/projects/:projectId/comparisons`) lacks ticket scoping
- Only 1 test file (though comprehensive at 358 lines)
- Most expensive ($19.78) with longest duration (54m)
