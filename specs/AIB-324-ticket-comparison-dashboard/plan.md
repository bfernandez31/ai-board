# Implementation Plan: Ticket Comparison Dashboard

**Branch**: `AIB-324-ticket-comparison-dashboard` | **Date**: 2026-03-20 | **Spec**: `specs/AIB-324-ticket-comparison-dashboard/spec.md`
**Input**: Feature specification from `/specs/AIB-324-ticket-comparison-dashboard/spec.md`

## Summary

Add database-backed storage for ticket comparison results and a rich visual comparison dashboard accessible from any participating ticket. The current file-based comparison system (markdown reports on branches) continues unchanged for backward compatibility. Three new Prisma models store structured comparison data, new API endpoints handle CRUD and enrichment (aggregating telemetry + quality scores), and new React components render a visual dashboard with ranking, metrics charts, decision points, and constitution compliance grid — all accessible via a conditional "Comparisons" tab on the ticket detail modal.

## Technical Context

**Language/Version**: TypeScript 5.6 (strict), Node.js 22.20.0
**Primary Dependencies**: Next.js 16 (App Router), React 18, Prisma 6.x, TanStack Query v5.90.5, Recharts 2.x, shadcn/ui
**Storage**: PostgreSQL 14+ via Prisma ORM — 3 new models (Comparison, ComparisonEntry, ComparisonDecisionPoint)
**Testing**: Vitest (integration + component), Playwright (E2E only if browser-required)
**Target Platform**: Web application (Next.js)
**Performance Goals**: Comparison dashboard loads within 2 seconds (SC-001)
**Constraints**: Backward compatibility with file-based markdown reports; no new dependencies beyond existing stack
**Scale/Scope**: Typically 2-5 tickets per comparison, up to ~50 comparisons per project

## Implementation Phases

### Phase 1: Data Layer (Database + Validation)

**Goal**: Persistent storage for comparison data

1. Add `Comparison`, `ComparisonEntry`, `ComparisonDecisionPoint` models to Prisma schema
2. Add reverse relations to `Ticket` and `Project` models
3. Run migration: `bunx prisma migrate dev --name add-comparison-models`
4. Create Zod validation schemas in `lib/schemas/comparison.ts`

**Acceptance**: `bunx prisma generate` succeeds; types available in Prisma client

### Phase 2: API Endpoints

**Goal**: CRUD operations for comparison data

1. **POST** `/api/projects/[projectId]/comparisons` — Save comparison (Bearer token auth, Zod validation)
2. **GET** `/api/projects/[projectId]/comparisons` — List project comparisons (session auth, paginated)
3. **GET** `/api/projects/[projectId]/comparisons/[comparisonId]` — Enriched detail (joins ticket metadata, telemetry, quality scores)
4. **GET** `/api/projects/[projectId]/tickets/[id]/comparisons/db` — List ticket's comparisons from DB
5. **GET** `/api/projects/[projectId]/tickets/[id]/comparisons/db/check` — Lightweight existence check

**Enrichment logic** (for GET detail):
- Join `Ticket` table for metadata (title, stage, workflowType, branch)
- Aggregate `Job` table for telemetry (sum costUsd, durationMs, tokens for COMPLETED jobs)
- Get latest COMPLETED verify Job's qualityScore/qualityScoreDetails
- Return null for missing telemetry/quality scores (UI handles graceful degradation)

**Acceptance**: All endpoints return correct responses; Zod rejects invalid payloads; auth enforced

### Phase 3: TanStack Query Hooks

**Goal**: Client-side data fetching layer

1. `useDbComparisonCheck(projectId, ticketId)` — staleTime: 30s (same as existing file-based check)
2. `useDbComparisonList(projectId, ticketId)` — staleTime: 30s
3. `useComparisonDetail(projectId, comparisonId)` — staleTime: 5m (comparisons are immutable)
4. `useSaveComparison()` — mutation with cache invalidation

**Acceptance**: Hooks fetch correct data; loading/error states handled

### Phase 4: UI Components

**Goal**: Visual comparison dashboard

1. **ComparisonDashboard** — Orchestrator: fetches detail data, renders 4 sections
2. **ComparisonRanking** — Cards per ticket: rank, score, agent, workflow type, winner highlight (green border/badge), key differentiators
3. **ComparisonMetrics** — Recharts horizontal `BarChart` for lines added/removed, source files, test files, test ratio; best value per metric highlighted
4. **ComparisonDecisions** — Collapsible sections per decision point: topic, verdict, per-ticket approach + assessment
5. **ComparisonCompliance** — Grid/table: rows = principles, columns = tickets; pass/fail badges with notes tooltip
6. **ComparisonListItem** — Clickable item showing date, participant count, winner, score

**Acceptance**: All sections render with mock data; winner highlighted; collapsibles work; graceful degradation for missing data

### Phase 5: Modal Integration

**Goal**: Access comparisons from ticket detail

1. Add `useDbComparisonCheck` call in ticket-detail-modal
2. Extend tab grid to support 5 columns (`grid-cols-5` when both hasJobs and hasComparisons)
3. Add conditional "Comparisons" `TabsTrigger` with comparison count badge
4. Add `TabsContent` rendering comparison list → dashboard navigation
5. Update keyboard shortcuts (Ctrl/Cmd+5 for Comparisons)

**Acceptance**: Tab appears only when comparisons exist; tab shows comparison list; clicking opens dashboard

### Phase 6: Command Integration

**Goal**: `/compare` command saves to DB after generating markdown

1. Update `ai-board.compare.md` to include API call step after markdown generation
2. Structure comparison output data to match `createComparisonSchema`
3. POST to `/api/projects/:projectId/comparisons` with Bearer token

**Acceptance**: Running `/compare` creates both markdown file and DB record

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Tab grid breaks with 5 columns on small screens | Use responsive grid: `grid-cols-3 md:grid-cols-5` with scroll on mobile |
| Enrichment query (joins across 3 tables) is slow | Use Prisma `select` to fetch only needed fields; telemetry aggregation uses indexed Job queries |
| Comparison data grows large (many decision points) | Decision points stored as separate rows, not single JSON blob; paginated APIs |
| Deleted tickets referenced in comparisons | ComparisonEntry retains captured metrics; UI shows "Ticket unavailable" with stored data |
