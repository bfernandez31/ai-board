## Enum Definition

### ClarificationPolicy

**Type**: PostgreSQL ENUM
**Purpose**: Define available clarification resolution strategies

**Values**:

| Value | Description | Use Case |
|-------|-------------|----------|
| `AUTO` | Context-aware (system default) | Detects keywords, applies CONSERVATIVE or PRAGMATIC based on context |
| `CONSERVATIVE` | Security & quality first | Payment, auth, PII, compliance, high-traffic features |
| `PRAGMATIC` | Speed & simplicity first | Admin tools, internal dashboards, prototypes, low-risk features |
| `INTERACTIVE` | Manual clarification (future) | Requires human input for ambiguities (preserves current behavior) |

### Effective Policy Resolution

**Algorithm**: Hierarchical fallback (ticket → project → system default)

**TypeScript Implementation**:
```typescript
// app/lib/utils/policy-resolution.ts
import { ClarificationPolicy, Ticket, Project } from '@prisma/client';

type TicketWithProject = Ticket & { project: Project };

export function resolveEffectivePolicy(
  ticket: TicketWithProject
): ClarificationPolicy {
  return ticket.clarificationPolicy ?? ticket.project.clarificationPolicy;
}

// Alternative: pure function for testing
export function resolveEffectivePolicyPure(
  ticketPolicy: ClarificationPolicy | null,
  projectPolicy: ClarificationPolicy
): ClarificationPolicy {
  return ticketPolicy ?? projectPolicy;
}
```

**Test Cases**:
```typescript
// test/unit/policy-resolution.test.ts
describe('resolveEffectivePolicy', () => {
  it('uses ticket override when present', () => {
    expect(resolveEffectivePolicyPure('CONSERVATIVE', 'AUTO'))
      .toBe('CONSERVATIVE');
  });

  it('uses project default when ticket is null', () => {
    expect(resolveEffectivePolicyPure(null, 'PRAGMATIC'))
      .toBe('PRAGMATIC');
  });

  it('uses project AUTO when both are default', () => {
    expect(resolveEffectivePolicyPure(null, 'AUTO'))
      .toBe('AUTO');
  });
});
```

## State Transitions

### Policy Lifecycle

**Project Policy**:
1. **Creation**: Project created → clarificationPolicy = AUTO (default)
2. **Update**: User changes policy via settings → clarificationPolicy updated
3. **Inheritance**: Tickets with null policy inherit from project

**Ticket Policy**:
1. **Creation**: Ticket created → clarificationPolicy = null (inherit from project)
2. **Override**: User sets ticket policy → clarificationPolicy = CONSERVATIVE/PRAGMATIC/AUTO
3. **Reset**: User clears ticket policy → clarificationPolicy = null (revert to project)
4. **Transition**: Ticket transitions to SPECIFY → effective policy used for spec generation

**No Retroactive Changes**:
- Changing project/ticket policy does NOT regenerate existing specs
- Only future SPECIFY transitions use new policy
- Existing specs preserve original Auto-Resolved Decisions section
