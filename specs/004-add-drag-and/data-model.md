# Data Model: Drag-and-Drop Ticket Movement

**Feature**: 004-add-drag-and
**Date**: 2025-10-01

## Overview

This feature requires modifications to the existing `Ticket` model to support optimistic concurrency control (version field) and alignment of stage names with the specification requirements.

## Entity: Ticket (Modified)


### Updated Schema
```prisma
enum Stage {
  INBOX   // renamed from IDLE
  PLAN    // unchanged
  BUILD   // unchanged
  VERIFY  // renamed from REVIEW
  SHIP    // renamed from SHIPPED
  // ERRORED removed (not in spec)
}

model Ticket {
  id          Int      @id @default(autoincrement())
  title       String   @db.VarChar(100)
  description String   @db.VarChar(1000)
  stage       Stage    @default(INBOX)

  @@index([updatedAt])
```

### Fields

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | Int | Primary Key, Auto-increment | Unique identifier |
| `title` | String | Max 100 chars, Not Null | Ticket title |
| `description` | String | Max 1000 chars, Not Null | Ticket description |
| `stage` | Stage (Enum) | Not Null, Default: INBOX | Current workflow stage |
| `version` | Int | Not Null, Default: 1 | Optimistic concurrency control version |
| `createdAt` | DateTime | Not Null, Auto-set | Creation timestamp |
| `updatedAt` | DateTime | Not Null, Auto-update | Last modification timestamp |

### Indexes

- **`stage`**: Fast filtering by column (board view)
- **`updatedAt`**: Fast sorting (most recent first)

### Validation Rules

**Stage Transition Rules** (enforced in application layer):
- Tickets can only move to the immediately next sequential stage
- Valid transitions: INBOXвҶ’PLAN, PLANвҶ’BUILD, BUILDвҶ’VERIFY, VERIFYвҶ’SHIP
- Invalid: Skipping stages (INBOXвҶ’BUILD), backwards movement (BUILDвҶ’PLAN)
- Terminal state: SHIP (no further transitions)

**Version Field Rules**:
- Incremented on every update (version++)
- Used for conflict detection in concurrent updates
- Client must send current version with update request
- Server rejects if version mismatch (409 Conflict)

## State Transitions

```
INBOX в”Җв”ҖвҶ’ PLAN в”Җв”ҖвҶ’ BUILD в”Җв”ҖвҶ’ VERIFY в”Җв”ҖвҶ’ SHIP
  вҶ‘        вҶ‘        вҶ‘         вҶ‘         вҶ‘
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
         (All backwards transitions INVALID)
```

**Sequential Progression Only**:
- Each stage can only transition to the immediately next stage
- No skipping stages
- No backwards movement
- SHIP is terminal (no further transitions)

## Concurrency Control Flow

```
Client Request:
  POST /api/tickets/123
  { stage: "PLAN", version: 1 }

Server Processing:
  1. Fetch ticket from database
  2. Check: ticket.version === request.version
     - If NO: Return 409 Conflict
     - If YES: Proceed
  3. Validate stage transition (INBOX вҶ’ PLAN)
     - If invalid: Return 400 Bad Request
     - If valid: Proceed
  4. Update atomically:
     UPDATE "Ticket"
     SET stage = 'PLAN', version = 2, updatedAt = NOW()
     WHERE id = 123 AND version = 1
  5. Return updated ticket

Client Handling:
  - Success (200): Update local state with new version
  - Conflict (409): Rollback optimistic update, show error
  - Invalid (400): Rollback optimistic update, show error
```

## Database Performance Considerations

### Query Patterns

**Read Pattern** (Board View):
```sql
SELECT id, title, stage, version, updatedAt
FROM "Ticket"
WHERE stage = ?
ORDER BY updatedAt DESC;
```
- Uses `stage` index for fast filtering
- Uses `updatedAt` index for fast sorting
- Efficient with <100 tickets per stage

**Update Pattern** (Drag-and-Drop):
```sql
UPDATE "Ticket"
SET stage = ?, version = version + 1, updatedAt = NOW()
WHERE id = ? AND version = ?
RETURNING id, stage, version, updatedAt;
```
- Atomic compare-and-swap using version
- Returns updated row for immediate response
- No explicit locking required (optimistic)

### Expected Load

- **Concurrent Users**: 10-50 simultaneous users
- **Update Frequency**: 1-5 updates/second during active use
- **Tickets per Stage**: <100 (spec requirement)
- **Total Tickets**: <500 (5 stages Г— 100 max)

### Performance Targets

- **Read Query**: <10ms (indexed, small dataset)
- **Update Query**: <20ms (atomic, single row)
- **Total API Latency**: <50ms (server processing)
- **Client-to-Visual**: <100ms (including optimistic update)
